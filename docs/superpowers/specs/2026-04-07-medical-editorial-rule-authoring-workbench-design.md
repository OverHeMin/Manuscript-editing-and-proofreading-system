# Medical Editorial Rule Authoring Workbench Design

**Date**

2026-04-07

**Goal**

为医学稿件处理系统 V1 设计一套可落地的规则录入工作台，让医学编辑能够把各类医学稿件规则结构化录入到统一规则库中，并让 `screening`、`editing`、`proofreading` 在调用时按稿件类型和期刊精准命中对应规则。

## Final Design Decisions

本次确认后的核心决策如下：

- 规则库采用双层模板体系：
  - `稿件类型大模板` 负责通用医学稿件规则
  - `期刊小模板` 负责期刊差异化覆盖规则
- 调用优先级固定为：`手工禁用/特批 > 期刊小模板 > 稿件类型大模板 > 系统默认规则`
- 规则源按模块复用：
  - `editing` 负责应用可自动执行的规则
  - `proofreading` 负责按同源规则做检查和报告
  - `screening` 负责做结构与合规性判断
- 规则录入端不采用一套空白通用表单，而采用：
  - `规则对象模板`
  - `结构化业务字段`
  - `系统匹配/执行字段`
- 第一阶段先做人工结构化录入
- 第二阶段再接入“从已校对稿件中 AI 提取规则候选 -> 人工确认 -> 入库”的候选录入入口
- 规则库是唯一真源
- 知识库只保存规则投影，用于检索增强、AI 理解和候选提取辅助，不直接决定规则执行

## Context

当前系统已经具备规则库的基础数据结构，但还不具备真正面向医学编辑的规则录入体验。

已确认的现状包括：

- 后端规则模型已经允许 `scope`、`trigger`、`action` 以结构化 JSON 承载扩展字段
- 现有前端规则录入字段偏少，主要集中在：
  - `scopeSections`
  - `scopeBlockKind`
  - `triggerKind`
  - `triggerText`
  - `actionKind`
  - `actionTarget`
- 当前确定性执行器仍以段落文本替换为主，只支持非常有限的自动格式动作
- 当前执行器尚未把 `表格` 当成一等对象，也尚未支持 Word 表格结构修改

这意味着：

- 规则存储层已经有扩展空间
- 但录入端和执行端都还停留在“段落/文本级规则”
- 对于医学稿件里高价值的规则资产，例如：
  - 结构化摘要标签
  - 统计学表达
  - 三线表
  - 参考文献格式
  - 伦理/注册/基金/利益冲突声明
  当前还没有一套适合医学编辑稳定录入和复用的工作台

## Problem Statement

当前版本存在 4 个核心问题：

1. 规则录入模型过薄
   无法表达医学稿件里的复杂规则对象，尤其是摘要、统计表达、表格、参考文献和期刊专属栏目。

2. 规则调用上下文不足
   当前规则体系还没有围绕“稿件类型 + 期刊 + 模块 + 文档对象”建立稳定的命中链路。

3. 规则对象缺少专属录入体验
   医学编辑需要录的是“三线表怎么排”“结构式摘要标签怎么写”“P 值和 CI 怎么写”，而不是底层 JSON。

4. 知识库与规则库关系需要明确
   需要既支持 AI 理解这些规则，又不能让 AI 检索结果直接取代结构化规则执行。

## Scope

本设计覆盖：

- 医学稿件规则库的分层结构
- 规则录入工作台的信息架构和交互流程
- 第一版应内置的规则对象模板
- 规则精准调用与覆盖策略
- 知识库中的规则投影设计
- 第一阶段与第二阶段的实施边界

本设计不覆盖：

- 全量自动表格改写算法的完整工程实现
- 第二阶段 AI 候选提取的完整 prompt 和模型细节
- 全量期刊模板内容建设
- 文档结构抽取引擎的所有低层实现细节

## Rule Library Architecture

### 1. 双层模板体系

规则库采用双层模板：

- `稿件类型大模板`
- `期刊小模板`

推荐的大模板集合：

- `clinical_study`
- `review`
- `systematic_review`
- `meta_analysis`
- `case_report`
- `guideline_interpretation`
- `expert_consensus`
- `diagnostic_study`
- `basic_research`
- `nursing_study`
- `methodology_paper`
- `brief_report`
- `other`

期刊小模板以具体期刊为单位建立，例如：

- `《中西医结合杂志》`
- `《中华医学杂志》`
- `《中国实用内科杂志》`

### 2. 模块复用

每套模板继续分为 3 个模块：

- `screening`
- `editing`
- `proofreading`

同一规则源在不同模块下执行不同动作：

- `screening`
  - 判断结构是否完整
  - 判断必要声明是否缺失
  - 判断关键栏目是否符合该稿件类型和期刊要求
- `editing`
  - 应用允许自动执行的规则
  - 对高风险规则输出候选修改或人工复核提示
- `proofreading`
  - 按同源规则检查并生成问题清单与说明

### 3. 继承与覆盖

调用链固定为：

1. 加载稿件类型大模板
2. 叠加期刊小模板
3. 由期刊小模板覆盖冲突规则
4. 再按模块和规则对象筛选出当前可用规则

冲突优先级固定为：

1. 手工指定禁用/特批规则
2. 期刊小模板
3. 稿件类型大模板
4. 系统默认规则

## Rule Objects

第一版规则录入端内置以下规则对象模板：

1. `题名与副题名`
2. `作者、单位、通信作者`
3. `摘要`
4. `关键词`
5. `正文标题层级`
6. `术语、缩略语、药名`
7. `数字、单位、时间、百分比`
8. `统计学表达`
9. `表格`
10. `图片、图注、流程图`
11. `参考文献`
12. `伦理、注册、知情同意、基金、利益冲突`
13. `稿件类型专属结构`
14. `期刊特有栏目`

### High-Value Examples

#### 摘要

摘要规则需要支持：

- 摘要类型：`非结构式` / `结构式`
- 结构标签集合：`目的/方法/结果/结论`
- 标签精确写法
- 括号样式
- 冒号样式
- 全角/半角规则
- 空格类型
- 字数范围
- 段落拆分方式
- 中英文摘要对应关系

必须支持精确示例：

- `摘要 目的 -> （摘要　目的）`

#### 表格

表格规则需要支持：

- 表格类型：`三线表` / `一般数据表` / `基线表` / `结局指标表`
- 表题位置与写法
- 表注位置与写法
- 线型规则
- 是否禁用竖线
- 表头层级
- 单元格对齐方式
- 数值小数位规则
- 缩略语和统计符号说明位置
- 跨页表处理要求
- 文中“见表”引用写法

必须把“三线表”作为第一批可录入对象，而不是把它折叠成自由文本备注。

#### 统计学表达

统计学表达规则需要支持：

- 统计学符号是否斜体
- `P` 值大小写与空格
- 置信区间写法
- `x±s` 或均数 ± 标准差写法
- `t`、`F`、`χ²`、`Z`、`r`、`OR`、`RR`、`HR` 的呈现规范
- 是否要求同时报告检验方法

## Authoring Workbench Information Architecture

### 1. 页面结构

规则录入工作台由 5 个区域组成：

1. `顶部上下文区`
   - 当前稿件类型大模板
   - 当前期刊小模板
   - 当前模块
   - 版本号
   - 状态：草稿 / 已发布 / 已归档

2. `左侧规则导航树`
   - 按规则对象分组，而不是按底层字段分组

3. `中间规则列表区`
   - 展示当前对象下的规则列表
   - 支持新建、复制、启用、停用、查看覆盖关系

4. `右侧结构化录入卡片`
   - 不同规则对象显示不同录入表单
   - 医学编辑无需直接面对原始 JSON

5. `底部验证与影响预览区`
   - 展示规则解释摘要
   - 展示潜在命中对象
   - 展示自动执行风险等级
   - 展示示例预览

### 2. 录入流程

单条规则录入流程固定为：

1. 选择 `稿件类型大模板`
2. 选择或创建 `期刊小模板`
3. 选择 `模块`
4. 选择 `规则对象`
5. 填写该对象专属录入卡片
6. 保存后查看“规则解释 + 示例预览 + 覆盖关系”

示例路径：

- `临床研究`
- `《中西医结合杂志》`
- `editing`
- `摘要`
- 填写标签写法：`摘要 目的 -> （摘要　目的）`

### 3. 录入字段分层

每条规则都必须具备两层字段。

#### 业务层字段

面向医学编辑：

- 规则名称
- 适用稿件类型
- 适用期刊
- 适用模块
- 规则对象
- 规则说明
- 正确示例
- 错误示例
- 是否允许自动执行
- 是否只做校对提示
- 证据级别

#### 系统层字段

面向匹配与执行：

- `scope`
- `selector`
- `trigger`
- `action`
- 冲突优先级
- 自动执行风险等级
- 手工复核模板

## Matching and Invocation

### 1. 调用上下文

每次进入 `screening`、`editing`、`proofreading` 前，系统先生成稿件上下文：

- 稿件类型
- 目标期刊
- 模块
- 文档结构
- 文档对象索引

文档对象索引至少包括：

- 章节
- 标题
- 段落
- 表格
- 表题
- 表注
- 图片与图注
- 参考文献条目
- 声明区块

### 2. 精准匹配依赖结构选择器

第一版选择器建议至少支持：

- `section_selector`
- `block_selector`
- `label_selector`
- `table_selector`
- `statement_selector`
- `pattern_selector`

示例：

- `section_selector=abstract`
- `block_selector=table`
- `table_selector.table_kind=three_line_table`
- `label_selector.text=摘要 目的`

### 3. 不同模块的读取方式

- `screening`
  - 重点读取“是否存在”“是否缺失”“是否合规”规则
- `editing`
  - 重点读取“可自动应用”“可候选应用”“需人工复核”规则
- `proofreading`
  - 重点读取“检查并报告”规则

## Knowledge Projection

规则库与知识库强关联，但职责不同：

- `规则库`
  - 唯一真源
  - 负责结构化存储
  - 负责调用、覆盖、执行、审计

- `知识库`
  - 保存规则投影
  - 负责检索增强
  - 负责 AI 理解与解释
  - 负责候选规则提取辅助

每条规则在知识库中的投影建议至少包含：

- 适用稿件类型
- 适用期刊
- 模块
- 规则对象
- 标准写法
- 常见错误
- 示例前后对照
- 场景说明
- 证据级别

原则上：

- 知识库帮助 AI 理解规则
- 规则库决定最终执行结果

## Implementation Staging

### Phase 1

先完成人工结构化录入能力：

- 规则录入工作台骨架
- 大模板与期刊小模板选择
- 规则对象导航
- 结构化录入卡片
- 规则解释与预览
- 规则覆盖关系展示

### Phase 1 High-Value Rule Objects

第一版优先做 6 组高价值对象：

1. `摘要`
2. `正文标题层级`
3. `数字、单位、统计学表达`
4. `表格`
5. `参考文献`
6. `伦理/注册/基金/利益冲突`

### Phase 2

接入候选规则提取入口：

- 从已校对稿件中提取规则候选
- 挂接到同一套规则对象模板
- 由医学编辑确认后入库

## Non-Goals for Phase 1

第一阶段不承诺：

- 所有表格规则都能端到端自动改写
- 所有期刊模板都一次性建设完成
- AI 无人工确认自动入库

第一阶段的目标是先把“规则录得准、调得准、看得懂”做出来。

## Acceptance Criteria

- 医学编辑可以按“稿件类型大模板 + 期刊小模板 + 模块 + 规则对象”的路径录入规则
- 系统可以保存结构化业务字段与系统字段
- 摘要规则可以表达 `摘要 目的 -> （摘要　目的）`
- 表格规则可以作为专属对象录入，而不是退化为自由备注
- 调用时可以按稿件类型、期刊、模块和结构选择器生成有效规则集
- `screening`、`editing`、`proofreading` 可以读取同一规则源，但执行各自模块化行为
- 知识库可保存规则投影，但不取代规则库作为执行真源

## Risks

### Risk 1

对象模板做得过于抽象，医学编辑仍然需要猜字段含义。

规避方式：

- 录入端以对象模板为中心
- 每个对象提供示例
- 业务字段优先于系统字段

### Risk 2

模板层级复杂，导致规则覆盖关系难以理解。

规避方式：

- 在规则详情中明确展示“来源层级”和“覆盖关系”
- 固定优先级，不允许隐式覆盖

### Risk 3

知识库与规则库职责混淆，导致 AI 检索结果直接参与执行。

规避方式：

- 保持“知识库负责理解，规则库负责执行”的边界
- 所有自动执行动作仍由规则库结构化规则驱动

### Risk 4

第一阶段过早承诺完整 DOCX 表格自动改写能力，导致实现风险过高。

规避方式：

- 第一阶段先把表格规则录入与读取打通
- 自动改写能力按对象逐步增强
