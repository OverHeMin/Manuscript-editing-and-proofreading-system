# Editorial Workbench UI Refresh Design

**Date**

2026-04-04

**Goal**

在不改变当前前端工作台架构、不引入新的大规模业务协议改造的前提下，为医学稿件处理系统 V1 建立一套可落地的系统级 UI 设计方案。该方案需要统一登录页、共享壳层、稿件处理工作台、知识审核工作台和治理配置页的视觉语言，并让系统形成稳定、专业、有识别度的“医学编辑部工作台”气质。

## Final Design Decisions

本次设计收敛后的核心决策如下：

- 系统总方向采用 `A2 再暖一点`
- 全系统配色基调采用 `B1 墨黑纸金`
- 校对模块不强行套用外部“问题卡片 + 锚点联动 + 建议栈”重交互方案
- 保留当前 `apps/web` 共享 workbench host 结构，不重做为新的路由壳层
- 黑金统一的是系统气质，而不是把全部正文工作区改成黑底

一句话定义：

`这是一个带有医学编辑部气质的黑金工作台系统，深色负责识别度与框架感，纸白负责正文可读性，温金负责强调与记忆点。`

## Scope

本设计覆盖：

- 登录页
- 共享 workbench 壳层
- 稿件工作台
- 知识审核工作台
- 治理配置页
- 系统级视觉语言
- 系统级组件语义
- 页面层级与黑金使用边界

本设计不覆盖：

- 新的 proofreading 专用重交互页面协议
- 新的 issue list / 文档锚点 / 实时建议联动 API
- 前端路由体系重构
- 业务流程重写
- 移动端或小程序端 UI 重设计

## Context

当前系统已经具备真实可用的工作台骨架，但视觉语言仍然偏通用后台：

- `apps/web/src/app/workbench-host.tsx` 负责共享 host、导航与工作台切换
- `apps/web/src/app/app.css` 提供全局壳层与登录卡基础样式
- `apps/web/src/features/manuscript-workbench/*` 承担稿件工作流主界面
- `apps/web/src/features/knowledge-review/*` 承担知识审核工作台
- `apps/web/src/features/admin-governance/*` 承担治理与配置页

当前架构的优点是：

- 工作台切换路径已经稳定
- 业务模块边界已经存在
- 页面大体布局已能支撑高密度工作流

当前问题是：

- 壳层与业务页之间缺少统一的视觉语义
- 登录、工作台、治理页看起来像同一套代码里的不同后台页面，而不是同一系统
- 稿件页、审核页、治理页没有形成清晰的页面家族层级
- 风格记忆点不足，系统辨识度低

## Problem Statement

如果直接实现外部校对模块的理想化 review UI，会引出两个问题：

1. 它与当前 workbench 架构不贴合，需要新的页面模型和更丰富的 proofreading API 合同。
2. 它会把大量设计资源投入到一个当前协议尚未稳定的子模块里，反而延后整个系统的 UI 一体化。

因此，这次设计不追求“先做一个最炫的校对页”，而是先解决更大的系统问题：

- 系统第一眼是否有明确身份
- 不同工作台是否属于同一套语言
- 高密度工作区是否既专业又不冰冷
- 后续新功能是否有统一视觉落点

## Design Principle

### 1. 风格服务工作流，不干扰工作流

视觉升级的目标不是把系统做得更花，而是让信息结构更清楚、系统气质更完整。

### 2. 黑金用于框架识别，不用于正文压迫

深色和金色承担品牌感、导航感、摘要感、强调感；正文处理区继续采用浅底和低干扰结构。

### 3. 稿件系统应像“编辑部”，不是“营销页”

系统应呈现专业、可信、克制、安静、有出版感的医疗编辑部氛围，而不是奢侈品官网或消费型产品。

### 4. 页面分家族，但不分世界观

登录页、稿件页、审核页、治理页可以有黑金浓度差异，但必须共享同一套色彩、圆角、边框、状态、摘要和标题语法。

### 5. 先统一系统 UI，再决定是否扩展校对模块交互

proofreading 的重交互设计留在后续单独规划，前提是新的数据契约和文档联动模式先稳定。

## Proposed Approach

### 1. 保留现有 workbench 架构，重做系统壳层与视觉语言

本次设计不改变：

- `workbench-host.tsx` 的共享入口模式
- 当前 hash-based workbench 切换方式
- 现有模块页的业务职责

本次设计改变：

- 全局色板与语义变量
- host 区的导航、上下文摘要、激活态、重点信息表达
- 业务页卡片、标签、摘要区、表单区、空状态、治理面板的风格语气

这意味着：

- 改动集中在 UI 层
- 风险主要是样式、布局和少量展示层结构调整
- 不要求引入新的业务 API

### 2. 建立“墨黑纸金”系统色板

推荐以 CSS 自定义变量建立系统色板，核心角色如下：

- `--color-shell-bg: #161514`
- `--color-shell-bg-strong: #1c1916`
- `--color-shell-text: #f0dfb8`
- `--color-accent-gold: #f0dfb8`
- `--color-accent-gold-muted: #8d7951`
- `--color-surface-paper: #f7f1e7`
- `--color-surface-card: #fffaf1`
- `--color-surface-card-strong: #fffdf8`
- `--color-border-warm: #d8c7aa`
- `--color-text-primary: #22344d`
- `--color-text-secondary: #5f6b78`

状态色建议：

- 成功：偏灰绿，不用高饱和亮绿
- 风险：偏砖红，不用刺眼正红
- 处理中：延续深色与金色系中的次强调色
- 待确认：以暖金和旧铜金表达

### 3. 将页面分成 5 个 UI 家族

#### 登录页

这是系统中黑金使用最完整的一页。

规则：

- 背景可以深色
- 品牌区可以使用更完整的墨黑 + 温金表达
- 表单卡本体必须保持浅底
- 主按钮使用深色底 + 金色或浅色文字

目标：

- 进入系统第一眼就建立“医学编辑部工作台”的身份感
- 保证登录错误、输入态和按钮清楚易读

#### 共享壳层

这是系统统一性的核心。

规则：

- 左侧导航为深色主骨架
- 顶部上下文摘要区可以深色
- 今日摘要、重点状态、当前角色和主入口形成稳定层级
- 主内容承载区保持纸白浅底

目标：

- 让不同业务页一进入就看起来属于同一套系统
- 先用壳层统一，再让业务页各自有轻微个性

#### 稿件工作台

这是最适合承接“编辑部温度”的业务页。

规则：

- 页面顶部摘要区可暖一些
- 阶段标签、推荐下一步、结果摘要允许更明显使用金色和深色强调
- 上传表单、资产链、作业结果、导出信息、明细摘要保持浅底
- 任何长文本、表单、文档信息都不采用深色大面积底板

目标：

- 让稿件页看起来最像编辑部工作区
- 保留操作清楚、上下文明确、摘要可扫描的特性

#### 知识审核工作台

这是秩序优先的审核页。

规则：

- 队列筛选头和动作区可以用深色
- 当前选中项用金色边界或暖色浅底突出
- 队列正文、详情、历史记录、备注区都保持浅底
- 视觉节奏比稿件页更克制

目标：

- 保持审核工作台的扫描效率
- 用少量黑金表达系统归属感，而不是牺牲阅读密度

#### 治理配置页

这是黑金浓度最低的一类页面。

规则：

- 仍然沿用统一壳层和头部风格
- 金色只作为分类、版本、发布提示和模块分组的强调
- 表单、列表、配置面板、证据区、审计信息全部保持浅底
- 版面应更规则、更文档化

目标：

- 让治理页显得稳、清楚、可审计
- 避免变成“运营活动页”或“展示型后台”

### 4. 建立系统组件语义

#### 按钮

- 主按钮：深色底，高对比浅字
- 次按钮：浅底纸白 + 暖边框
- 风险按钮：低饱和砖红
- 幽灵按钮：边界清楚但存在感弱

#### 卡片

- 默认卡片使用暖白纸色
- 边框为低对比暖灰线
- 阴影轻、短、低漂浮
- 首页说明卡、空状态卡允许更明显纸感
- 表格和表单周边卡片保持克制

#### 标签与状态徽章

- 统一使用胶囊形或短矩形圆角标签
- 标签优先高可读，不使用手写体
- 审核状态、流程状态、证据级别使用同一标签体系

#### 表格与队列

- 表头与工具栏可以承接部分风格
- 表格主体不加入纸纹或装饰图案
- 行 hover 只做轻微底色变化
- 选中态使用细边框、弱高亮或左侧强调条

#### 表单

- 所有输入框保持浅底
- 激活态以金色或暖色描边强调
- 不采用高亮发光、强渐变、厚阴影

#### 抽屉与右侧详情板

- 标题区可承接少量深色或金色模块识别
- 主体区仍以浅底强可读排版为主
- 底部操作区应固定且层级清楚

#### 空状态与引导说明

- 这是最适合承接系统情绪和插画气质的区域
- 可以加入更暖的编辑部语气
- 行动按钮仍要明确，不被装饰性元素稀释

### 5. 排版与动效

#### 排版

- 标题要有出版感，但只用于欢迎区、页面抬头、空状态、模块头部
- 正文、表格、表单、证据、日志必须使用稳妥高可读字体
- 数字与状态文本保持规整，不做夸张字形处理

#### 动效

- 页面切换：淡入 + 轻上浮
- 卡片与摘要：轻微错峰出现
- hover：1px-2px 的轻微上浮或提亮
- 禁止闪烁、强弹跳、强发光、复杂粒子效果

## Architecture Impact

本设计对当前系统架构的影响控制在展示层，原则上不改变业务结构。

### 不改变的部分

- workbench host 的组织方式
- hash-based navigation
- 模块页之间的职责边界
- proofreading 的现有 API 契约
- knowledge review 的现有双栏工作台结构

### 需要改动的部分

- `apps/web/src/app/app.css`
- `apps/web/src/app/workbench-host.tsx`
- 业务页对应 CSS 文件
- 少量页面头部与摘要层结构

### 可能新增的整理项

- 统一全局 design tokens
- 共用 card / badge / section header / page summary 样式语义
- 对现有页面中的标题和摘要结构做轻度标准化

## Rollout Strategy

建议按风险最低的顺序推进：

### Phase 1. 系统壳层与设计 token

- 先改全局色板、圆角、边框、阴影与字体层级
- 改造登录页与共享壳层
- 让系统先获得统一身份

### Phase 2. 稿件工作台页面家族

- 优先改顶部摘要、结果卡、推荐下一步、表单面板
- 保留当前业务动作结构

### Phase 3. 知识审核工作台页面家族

- 统一队列头、详情头、动作区、选中态
- 保持现有双栏结构与审核动作流

### Phase 4. 治理配置页页面家族

- 把治理页收进统一黑金体系
- 降低暖感，加强秩序感与规则感

### Phase 5. 评估是否需要单独规划 proofreading review UI

- 仅在 proofreading 的数据契约稳定后再评估
- 不在本轮系统 UI 升级中强行插入

## Risks

### Risk 1. 黑金使用过重，影响阅读与表单效率

规避方式：

- 明确“深色只用于壳层、摘要层、强调层”
- 所有高密度正文区保持浅底

### Risk 2. 登录页、稿件页、治理页各自漂亮，但不像一个系统

规避方式：

- 先做系统 token 与共享壳层
- 再做页面家族差异，而不是反过来

### Risk 3. 为了贴合外部校对概念而引入不必要的 API 改造

规避方式：

- 本轮禁止把外部校对概念硬塞进现有 UI
- proofreading 后续单独建模、单独评估

### Risk 4. 治理页被做得过暖，失去规则平台气质

规避方式：

- 治理页使用同源但更冷静的黑金表达
- 限制金色在治理页的使用面积

## Acceptance Criteria

- 用户进入登录页即可感知明确的“医学编辑部黑金工作台”身份
- 共享壳层能够统一所有工作台的视觉世界观
- 稿件工作台比当前版本更有编辑部气质，但不降低可读性
- 知识审核工作台保留审核效率，同时纳入统一黑金系统
- 治理配置页维持冷静、规则化、可审计的表达
- 全系统形成统一的按钮、卡片、标签、摘要、空状态语义
- 本轮设计不要求新增 proofreading 富交互协议

## Implementation Notes For Planning

下一阶段写实现计划时，应该以“壳层优先、业务页分家族推进”的方式拆任务，而不是按单个页面零散修补。推荐的实现计划拆分如下：

1. 全局 token 与共享壳层
2. 登录页与 auth 视图
3. 稿件工作台视觉收敛
4. 知识审核工作台视觉收敛
5. 治理配置页视觉收敛
6. 系统级回归与响应式检查
