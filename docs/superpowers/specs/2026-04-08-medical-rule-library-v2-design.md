# Medical Rule Library V2 Design

**Date**

2026-04-08

**Status**

Draft for review

**Goal**

在规则库 V1 已经完成“双层模板、规则真源、知识投影、编辑/校对共源调用”的基础上，建设规则库 V2：把规则库升级为一个可长期运营的“全对象统一规则录入与候选学习一体化平台”，让医学编辑团队能够稳定录入、审核、调用、学习和迭代各类医学稿件规则。

## Final Design Decisions

本次确认后的 V2 核心决策如下：

- V2 不做成“更大的规则表单”，而做成“规则生产平台”。
- V2 在产品上保持一个 `规则中心`，但内部明确分为两个工作台：
  - `规则录入工作台`
  - `规则学习工作台`
- V2 直接按完整目标规划，不再拆成“只做录入框架”的缩水版本。
- V2 的主交互采用：
  - `混合式入口`
  - `业务录入优先`
  - `高级区按需展开`
- V2 一次性覆盖全部核心规则对象，而不只做少数几个高价值对象。
- V2 同时纳入“从已校对稿件中 AI 抽取候选规则 -> 人工确认 -> 入库”的学习链路。
- 规则库仍然是唯一真源；知识库仍然只保存规则投影，不直接决定执行。
- 调用优先级继续保持：
  - `手工禁用/特批 > 期刊小模板 > 稿件大模板 > 系统默认`
- V2 把“命中解释、冲突解释、预演验证、覆盖关系可视化”视为一等能力，而不是附加能力。
- V3 不再承担核心治理功能，只承担自动化增强、效果分析和规模化优化能力。

## Context

规则库 V1 已经完成第一版落地，当前具备的能力包括：

- `editing` 与 `proofreading` 基于同一规则源进行处理
- 稿件大模板与期刊小模板的双层模板结构
- 规则真源与知识投影的职责分离
- 规则对象基础结构、期刊覆盖和运行时规则解析
- 规则录入工作台第一版

但 V1 仍然主要解决了“规则能存、能调、能初步录”的问题，尚未彻底解决以下问题：

1. 录入体验仍然偏工程化
   医学编辑仍然容易感到字段过碎、对象体验不够强、批量维护能力不足。

2. 规则对象覆盖仍不够完整
   尽管已经支持高价值对象，但仍未形成“所有核心医学规则对象都能统一录入”的完整体系。

3. 规则学习闭环尚未正式接入
   目前还没有把“从已校对稿件中发现规则 -> 人工审核 -> 正式入库”形成稳定平台能力。

4. 命中解释与冲突治理还不够产品化
   规则为什么命中、为什么覆盖、为什么不执行，还没有形成足够清晰的解释系统。

因此，V2 的任务不是简单扩字段，而是把规则库升级为长期可运营的规则平台。

## Problem Statement

V2 需要系统性解决以下 6 类核心问题：

1. `如何让医学编辑录规则更顺手`
   要从“底层字段录入”转向“对象化、业务化、批量化录入”。

2. `如何覆盖全部医学核心规则对象`
   规则库不能只擅长摘要和少量文本替换，还要覆盖表格、统计学表达、声明、参考文献等。

3. `如何把学习能力接进来`
   已校对稿件里的高价值经验不能长期停留在人工脑中，必须进入候选规则审核链。

4. `如何让命中更精准`
   运行时必须按稿件类型、期刊、模块、对象和结构选择器稳定命中正确规则。

5. `如何让规则冲突可解释`
   必须明确显示规则来源、覆盖关系、优先级和是否自动执行。

6. `如何保证后续 V3 风险显著下降`
   V2 必须先把平台底座做稳，V3 才能安全做自动化增强和效果优化。

## Scope

本设计覆盖：

- 规则中心 V2 总架构
- 双工作台产品模型
- 全对象规则模型
- 统一录入框架
- 候选规则学习与人工审核链
- 命中、覆盖、冲突解释与预演验证
- 知识库投影升级
- V2 与 V3 的分工边界

本设计不覆盖：

- 所有复杂表格和图文对象的端到端自动排版实现细节
- 候选规则抽取模型的具体 prompt 与模型参数
- 全量 BI 看板与统计报表的最终图表方案
- 自动优化策略的精细算法实现

## V2 Platform Architecture

V2 建议采用四层平台结构：

### 1. Rule Source Layer

规则真源层，负责保存：

- 稿件大模板
- 期刊小模板
- 规则对象
- 规则版本
- 规则状态
- 覆盖关系
- 审核轨迹
- 来源方式
- 命中解释元数据

这一层是唯一真源，任何正式调用都必须从这里读取。

### 2. Rule Authoring Layer

规则录入层，负责：

- 手工新建规则
- 规则复制与批量维护
- 模板层级选择
- 对象化录入
- 高级参数精修
- 预演验证
- 提交审核与发布

### 3. Rule Learning Layer

规则学习层，负责：

- 从历史稿件中提取候选规则
- 候选规则归类
- 相似规则比对
- 人工审核
- 确认入库
- 驳回、合并、转知识投影

### 4. Rule Consumption Layer

规则消费层，负责为以下模块提供统一规则源：

- `screening`
- `editing`
- `proofreading`
- 知识投影与 AI 检索增强

## Product Model: One Center, Two Workbenches

V2 产品上建议保留一个统一的 `规则中心`，但在内部明确分成两类工作台。

### A. Rule Authoring Workbench

面向“我要构建、维护、发布正式规则”的场景。

主要能力：

- 向导式逐条录入
- 表格式批量维护
- 对象专属录入卡
- 高级字段展开
- 命中预演
- 覆盖关系查看
- 版本、状态与审核流

### B. Rule Learning Workbench

面向“系统发现了疑似规则，请人工审核”的场景。

主要能力：

- 候选规则队列
- 原文与修改后对照
- 规则抽取解释
- 相似规则比对
- 一键转正式规则草稿
- 合并 / 驳回 / 转知识说明

### Why Two Workbenches

采用双工作台而不是单大页面的原因：

- 录入规则和审核候选规则是两种完全不同的工作心智
- 医学编辑团队通常会形成不同分工
- 候选审核链会持续增长，如果混入正式录入工作台，会显著污染主操作流
- 后续 V3 强化学习能力时，不需要重做录入台

## Rule Object Panorama

V2 直接按全对象设计，不再只围绕少数规则对象优化。

建议一次性纳入以下对象：

1. 标题与副标题
2. 作者、单位、通信作者
3. 摘要
4. 关键词
5. 正文标题层级
6. 术语、缩略语、药名
7. 数字、单位、时间、百分比
8. 统计学表达
9. 表格
10. 图片、图注、流程图
11. 参考文献
12. 伦理、注册、基金、利益冲突声明
13. 稿件类型专属结构
14. 期刊专属栏目

其中高价值、高优先对象包括：

- 结构式摘要
- 三线表
- 统计学表达
- 参考文献
- 伦理与注册声明

## Unified Rule Model

每条规则在 V2 中建议由 6 组字段组成：

### 1. Applicability Fields

- 适用稿件大模板
- 适用期刊小模板
- 适用模块
- 适用规则对象
- 是否覆盖上层规则

### 2. Business Fields

面向医学编辑的结构化业务字段。

例如：

- 摘要标签集合
- 括号样式
- 表格类型
- P 值写法
- 参考文献格式要求
- 伦理声明标准措辞

### 3. Runtime Fields

面向系统命中与执行：

- `scope`
- `selector`
- `trigger`
- `action`
- 排除条件
- 风险等级
- 自动执行策略

### 4. Explanation Fields

面向 AI 理解和人工复核：

- 规则解释
- 正误示例
- 适用场景
- 不适用边界
- 审核提示文本

### 5. Governance Fields

- 规则版本
- 状态
- 创建人
- 审核人
- 来源方式
- 证据级别
- 生效时间

### 6. Linkage Fields

- 关联知识投影
- 关联候选规则来源
- 关联历史稿件证据
- 关联被覆盖 / 覆盖对象

## Rule Authoring Workbench Information Architecture

录入工作台建议采用 `一中心、三入口、四视图`。

### One Center

- `规则中心`

### Three Entrances

1. `新建规则`
2. `批量维护`
3. `候选入库`

### Four Views

1. `上下文栏`
   - 大模板
   - 期刊小模板
   - 模块
   - 当前对象
   - 版本状态

2. `对象导航栏`
   - 按规则对象分组
   - 按对象显示规则数量、草稿数量、候选数量

3. `规则列表台账`
   - 展示当前对象下的规则
   - 显示来源、状态、覆盖关系、更新时间、执行方式

4. `右侧编辑面板`
   - 默认业务录入
   - 可展开高级区

## Authoring Interaction Modes

### Guided Authoring

用于新增规则。

建议流程固定为：

1. 选择稿件大模板
2. 选择期刊小模板
3. 选择模块
4. 选择规则对象
5. 填写对象专属录入卡
6. 系统自动生成解释、命中条件和预览
7. 保存草稿
8. 预演命中
9. 提交审核或发布

### Grid Maintenance

用于批量维护。

建议支持：

- 按稿件类型筛选
- 按期刊筛选
- 按模块筛选
- 按对象筛选
- 按状态筛选
- 按来源方式筛选
- 批量启停
- 批量复制
- 批量提交审核
- 导出清单

### Advanced Editing

默认隐藏高级区，仅在需要时展开。

高级区至少包括：

- selector
- trigger
- action
- 命中优先级
- 自动执行风险等级
- proofreading 检查口径
- AI 理解说明
- 排除条件
- 手工复核提示

## Object-Specific Authoring Cards

V2 必须为每类对象提供专属录入卡，而不是复用一个“万能表单”。

### Abstract Card

应支持：

- 摘要类型
- 标签集合
- 标签顺序
- 括号样式
- 冒号样式
- 全角 / 半角规则
- 空格类型
- 中英文对应
- 字数范围

必须支持精确示例：

- `摘要 目的 -> （摘要　目的）`

### Table Card

应支持：

- 表格类型
- 三线表要求
- 竖线禁用规则
- 表题位置
- 表注样式
- 表头层级
- 数值对齐
- 跨页处理
- 缩略语与统计符号说明位置

### Statistics Card

应支持：

- P 值写法
- CI 写法
- 统计符号斜体规则
- 均数 ± 标准差写法
- OR / RR / HR 表达
- 检验方法是否必须说明

### Statement Card

应支持：

- 是否必须出现
- 推荐栏目位置
- 标准表述
- 缺失时检查规则
- 变体写法映射

## Rule Learning Workbench Design

规则学习工作台的原则是：

`AI 负责发现，人工负责确认，规则库负责生效`

### Candidate Sources

V2 建议纳入以下候选规则来源：

- 已校对完成稿件
- 编辑前后对照稿件
- proofreading 人工修订结果
- 团队手工上传规则样本
- 已确认知识条目中的规范片段

### Candidate Review Card

每个候选规则需要展示：

- 候选规则标题
- 大模板 / 小模板 / 模块 / 对象
- 原文片段
- 修改后片段
- AI 归纳解释
- 建议 selector / trigger / action
- 相似现有规则
- 差异点
- 风险等级
- 推荐动作

### Candidate Actions

审核人至少可以执行：

- 采纳为新规则
- 合并到现有规则
- 作为期刊覆盖规则采纳
- 仅保存为知识说明
- 驳回
- 转交复核

### Authoring-Learning Linkage

录入台与学习台应强联通：

- 学习台点击“采纳”后进入录入台预填表单
- 正式规则详情可以反查候选来源
- 候选卡可以显示已存在规则是否已吸收该经验

## Matching and Invocation

V2 命中链建议固定为：

1. 识别稿件上下文
2. 确定稿件大模板
3. 确定期刊小模板
4. 确定模块
5. 建立文档对象索引
6. 按对象筛规则
7. 按优先级决策
8. 输出命中解释

### Document Object Index

文档对象索引至少覆盖：

- 章节
- 标题
- 段落
- 摘要标签
- 表格
- 表题
- 表注
- 图题
- 声明区块
- 参考文献条目

### Resolution Priority

规则优先级保持为：

1. 手工禁用 / 手工特批
2. 期刊小模板规则
3. 稿件大模板规则
4. 系统默认规则

同层冲突时再比较：

- 对象精度
- selector 具体度
- 替代声明
- 风险策略

不采用“谁新谁赢”的隐式逻辑。

## Explainability and Conflict Governance

V2 每条生效规则都应能回答：

1. 为什么命中了它
2. 为什么没有命中别的相似规则
3. 它覆盖了谁
4. 它是自动执行还是仅提示

建议产品层提供规则解释卡：

- 当前对象
- 命中规则
- 命中原因
- 覆盖来源
- 执行动作
- 风险等级
- 是否需要人工复核

## Preview and Validation

V2 建议把“预演验证”作为正式发布前的必经步骤。

至少支持 3 种预演方式：

### 1. Single Rule Preview

给定一段文本或对象，测试单条规则的命中与输出。

### 2. Object Bundle Preview

针对摘要、表格、参考文献等对象，测试当前模板下全部规则的综合效果。

### 3. Historical Manuscript Replay

拿历史稿件进行整体验证，查看：

- 命中了哪些规则
- 漏掉了哪些规则
- 是否有误伤
- 哪些规则发生冲突

## Knowledge Projection

知识库仍不是执行真源，但在 V2 中应升级为“规则理解投影层”。

每条规则投影到知识库时建议保存：

- 规则对象
- 适用稿件类型
- 适用期刊
- 标准写法
- 常见错误写法
- 正误示例
- 适用场景说明
- 不适用边界
- 来源证据
- 是否仅作理解、不作执行

知识库职责：

- 帮助 AI 理解规则
- 帮助 screening 做检索辅助
- 帮助 editing / proofreading 形成更稳定解释
- 帮助候选规则学习阶段做相似比对

规则库职责：

- 决定最终执行
- 决定优先级与覆盖关系
- 决定自动执行与人工复核策略

## V2 Delivery Strategy

本次建议把 V2 作为一个完整平台目标一次立项，但不要求所有能力同一天全量开放。

### Recommended Strategy

采用：

- `一次性规划完整 V2`
- `分批开闸上线能力`

原因：

- 满足“能力规划一步到位”
- 降低单次上线风险
- 保持架构与状态机从第一天起按完整形态设计

### Suggested Internal Activation Order

1. 统一录入框架与全对象模型
2. 批量维护、覆盖关系、命中预演
3. 候选规则审核入口
4. 自动候选抽取与审核入库
5. 效果分析和回归运营能力

## V2 vs V3 Boundary

### V2 Solves

- 规则能被稳定生产
- 规则能被稳定审核
- 规则能被稳定调用
- 规则能被稳定学习
- 规则冲突可解释
- 规则可通过知识投影服务 AI 理解

### V3 Focuses On

在 V2 底座稳定后，V3 再承担优化型能力：

- 候选规则聚类与去重
- 命中率 / 误报率看板
- 自动优化建议
- 跨期刊规则继承建议
- 批量历史稿件规则挖掘
- 更强的复杂表格与版式自动改写

V2 解决的是“规则治理平台能不能成立”，V3 解决的是“规则治理平台能不能更聪明、更高效、更大规模化”。

## Acceptance Criteria

- 医学编辑能够按 `稿件大模板 + 期刊小模板 + 模块 + 规则对象` 的路径录入任何核心规则对象
- 录入工作台同时支持向导式录入与表格台账维护
- 业务字段优先、技术字段可展开的交互模型成立
- 所有核心对象具备专属录入卡
- 学习工作台能够接收 AI 候选规则并支持人工确认入库
- 正式规则能够展示命中解释、覆盖关系与自动执行策略
- 发布前支持单规则预演、对象预演与历史稿件回放
- `screening`、`editing`、`proofreading` 继续基于同一规则真源调用
- 知识库只保存规则投影，不替代规则库成为执行真源

## Risks

### Risk 1

一次性纳入全部对象后，页面结构变重，编辑难以上手。

**Mitigation**

- 用对象导航和向导式录入降低复杂度
- 批量维护与逐条录入拆入口
- 高级区默认隐藏

### Risk 2

候选规则学习链加入后，低质量候选过多，审核压力上升。

**Mitigation**

- 设置候选入队门槛
- 相似规则聚合显示
- 支持批量驳回与转知识说明

### Risk 3

覆盖关系过多后，规则解释成本上升。

**Mitigation**

- 固定优先级体系
- 明确替代与被替代链
- 在 UI 中直接显示命中解释卡

### Risk 4

团队急于把所有自动化能力同时放开，导致上线风险增高。

**Mitigation**

- 一次规划完整能力，但分批开闸
- 先稳住录入、命中、审核和预演
- 自动学习与复杂自动改写按开关管理

## Summary

规则库 V2 不再是规则录入功能的延伸，而是面向医学编辑团队的正式规则治理平台。

它要同时解决：

- 全对象统一录入
- 人工与 AI 协同学习
- 运行时精准命中
- 冲突与覆盖可解释
- 知识投影服务 AI 理解

只要 V2 这层底座成立，后续 V3 的风险会显著下降，因为 V3 将主要做优化和智能增强，而不再需要反复改动规则平台的核心结构。
