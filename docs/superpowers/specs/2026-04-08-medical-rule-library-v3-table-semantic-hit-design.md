# Medical Rule Library V3 Table Semantic Hit Design

**Date**

2026-04-08

**Status**

Draft for written review

**Goal**

在规则库 V2 已经完成“规则中心、规则真源、知识投影、编辑与校对共源调用”的基础上，启动规则库 V3 第一阶段：聚焦 `DOCX / Word` 医学稿件中的三线表，把原始 Word 表格结构升级为可命中的表格语义层，让 `editing` 与 `proofreading` 都能基于同一套表格语义对象稳定命中规则，并支持“大模板通用规则 + 期刊小模板覆盖规则”的可解释执行。

## Final Design Decisions

本次确认后的 V3 第一阶段核心决策如下：

- V3 第一阶段只做 `DOCX / Word` 输入，不覆盖 PDF 和其他来源格式。
- V3 第一阶段聚焦 `医学稿件中的三线表`，不扩展到图片型表格、图表、流程图。
- V3 第一优先不是“自动排版更漂亮”，而是“结构位置识别更准确”。
- V3 采用 `语义分层路线`，不采用“继续堆字符串匹配”或“直接增强原始 selector”的路线。
- V3 先建立 `通用医学三线表语义层`，再提供 `期刊小模板覆盖钩子`。
- `editing` 与 `proofreading` 必须消费同一套表格语义层，避免两个模块各自理解表格。
- 规则库仍然是唯一执行真源；知识库仍然只保存规则投影与理解增强，不直接决定表格规则执行。
- V3 第一阶段不承担：
  - 全自动三线表排版美化
  - 所有异常表格的一次性全覆盖
  - 历史稿件自动学习闭环增强
  - BI 级命中分析看板

## Context

规则库 V2 已经完成以下能力基础：

- `editing` 与 `proofreading` 基于同一规则源执行与检查
- 大模板与期刊小模板的双层模板结构
- 规则中心中的规则录入与规则学习工作台
- 规则对象目录、规则投影、知识投影与学习写回链路
- 规则命中解释、覆盖解释和治理流程的第一版能力

但 V2 之后仍然存在一个关键准确率问题：

`表格规则已经可以录入，但系统还不能稳定看懂表格内部结构，因此规则经常难以精准命中到正确位置。`

这在医学稿件里尤其突出，因为三线表常常同时具备以下特点：

- 多层表头
- 合并单元格
- 左侧项目列与普通数据列语义不同
- 单位信息可能落在列头、项目名或表注
- 表注里同时出现统计学脚注、缩略语解释和显著性说明

如果仍然直接对 Word 原始表格坐标、原始 XML 结构或脆弱的文本片段写规则，后果通常是：

- 同一条规则在不同稿件中命中不稳定
- 规则覆盖关系变得难解释
- 编辑与校对无法共享稳定的目标位置
- 后续要做自动修正或更复杂期刊规则时返工风险很高

因此，V3 第一阶段要先补的是“表格语义底座”，而不是继续堆更多表格规则。

## Problem Statement

V3 第一阶段需要系统性解决以下 4 类问题：

1. `系统如何先看懂表`
   在规则执行前，系统必须把 Word 表格先还原成稳定的医学表格语义结构，而不是直接对原始行列坐标匹配。

2. `规则如何打到正确位置`
   规则必须面向 `header_cell / data_cell / footnote_item / unit_marker` 等语义对象编写与命中，而不是依赖固定“第几行第几列”。

3. `期刊覆盖如何正确落位`
   大模板与期刊小模板必须作用于同一套表格语义对象，期刊覆盖只改变最终生效规则，不重新理解整张表。

4. `命中原因如何可解释`
   对运营人员、医学编辑和审核人员来说，系统必须清楚解释：
   - 命中了哪一张表
   - 命中了哪个语义对象
   - 为什么命中
   - 命中的是通用规则还是期刊覆盖规则
   - 最终执行了哪条规则

## Scope

### In Scope

- `DOCX / Word` 医学稿件中的三线表识别
- 表格内部结构语义化
- 表格语义层输出与持久化或运行时传递
- 基于语义对象的规则 selector / 命中机制
- 大模板表格通用规则命中
- 期刊小模板表格覆盖命中
- `editing` 使用语义层执行表格规则
- `proofreading` 使用语义层检查并报告表格规则
- 命中解释与覆盖解释
- 代表性医学三线表 fixture、回归测试与验收样例

### Out of Scope

- PDF 表格识别
- 非 Word 来源的复杂表格统一解析
- 图片型表格与图表对象理解
- 表格全自动美观排版
- 所有异常版式的一次性全面修复
- 从历史稿件自动抽取新表格规则的学习闭环增强
- 大规模命中分析 BI 看板

## V3 Architecture Overview

V3 第一阶段建议采用如下链路：

`Word 原始表格 -> 表格语义层 -> 通用表格规则命中 -> 期刊覆盖规则命中 -> 编辑执行 / 校对报告 -> 命中解释`

### 1. Table Semantic Extraction Layer

文档管线先把 Word 中的原始 `table` 解析为稳定的表格语义对象集合。这一层不直接做规则决策，只负责“把表看懂”。

### 2. Table Semantic Runtime Layer

把表格语义对象以稳定 contract 挂入编辑与校对可消费的运行时上下文。后续命中、报告、解释都基于这层对象。

### 3. Rule Resolution Layer

规则解析顺序保持与系统整体规则架构一致：

- 先命中大模板中的通用医学表格规则
- 再命中期刊小模板中的表格覆盖规则
- 由期刊层决定最终生效规则

### 4. Explainability Layer

把命中证据、覆盖来源、最终执行规则回写给工作台与报告输出，使运营人员能理解整条命中链路。

## Core Semantic Model

V3 第一阶段建议把一张三线表拆解为以下稳定语义对象：

### 1. `table_block`

整张表的总容器，负责承载：

- 表 ID
- 在稿件中的顺序
- 所属稿件模块
- 来源段落范围
- 表格语义摘要

### 2. `table_label`

表号对象，例如：

- `表1`
- `表2`

### 3. `table_title`

表题正文对象，例如：

- “不同治疗组基线特征比较”

### 4. `header_zone`

表头区域对象。它不是简单的“第一行”，而是一块区域，用来容纳多层表头。

### 5. `header_cell`

表头单元格对象。每个 `header_cell` 至少需要保留：

- 文本内容
- 行列位置
- 跨行关系
- 跨列关系
- 父级 / 子级表头路径
- 归属列语义

### 6. `stub_column`

左侧项目列对象。医学稿件中的“组别”“观察指标”“项目”“时间点”等常常位于该区域，它与普通数据列必须分开处理。

### 7. `data_region`

数据区对象，表示真正承载数值和正文表格内容的区域。

### 8. `data_cell`

数据区单元格对象。每个 `data_cell` 至少需要能回答：

- 对应哪个行项目
- 对应哪个列表头
- 是否带单位上下文
- 是否疑似统计学结果位
- 是否与注释符号关联

### 9. `unit_marker`

单位标记对象。单位可能来自：

- 列头
- 左侧项目名
- 表题
- 表注

单位不能继续只当普通字符串处理，而应成为独立语义对象或独立属性。

### 10. `footnote_zone`

表注区域对象。

### 11. `footnote_item`

具体表注项对象，例如：

- `a`
- `b`
- `*P<0.05`
- 缩略语解释

### 12. `merged_relation`

合并关系对象，用来保存：

- 哪些单元格逻辑上属于同一语义块
- 哪些表头是跨列或跨行形成
- 哪些数据区与表头存在对齐依赖

### 13. `table_semantic_profile`

整表语义摘要，用于快速命中和后续解释，例如：

- 是否为典型三线表
- 是否多层表头
- 是否存在合并表头
- 是否包含统计列
- 是否包含单位混排
- 是否包含脚注符号

## Semantic Coordinates

为了让规则稳定命中，表格语义对象需要输出比“原始行列号”更稳定的语义坐标。建议至少支持：

- `table_id`
- `zone_type`
- `cell_id`
- `header_path`
- `row_key`
- `column_key`
- `unit_context`
- `footnote_anchor`
- `merged_group_id`

后续规则 selector 应优先命中这些语义坐标，而不是直接命中 Word 原始坐标。

## Rule Hit Model

V3 第一阶段的命中逻辑建议明确为：

### Step 1: Parse Before Match

先解析出表格语义对象，再进入规则命中。

原则是：

`先理解表，再套规则`

### Step 2: Match Generic Medical Rules First

大模板中的通用医学规则优先命中。例如：

- 三线表不允许竖线
- 表题应位于表上方
- 单位应统一放在列头或项目名后
- 统计学表达应与正文规则一致
- 表注中的显著性说明必须标准化

这类规则应面向语义对象编写，例如：

- 作用对象：`header_cell`
- 条件：`contains_unit_marker = true`
- 检查点：`unit_format`

或：

- 作用对象：`footnote_item`
- 条件：`looks_like_statistical_note = true`
- 检查点：`p_value_style`

### Step 3: Apply Journal Overrides

当用户在编辑或校对前选择了某个期刊小模板时，期刊规则在同一套语义对象上执行覆盖。

也就是说：

`同一张表 -> 同一套语义对象 -> 先跑通用规则 -> 再跑期刊覆盖`

而不是重新解析一遍表，或单独为期刊层建立另一套命中逻辑。

### Step 4: Produce Operator-Readable Explanation

系统必须输出可读的命中解释，例如：

- `表2 -> header_cell[治疗组] -> 命中通用规则：单位写法检查`
- `后续被《中西医结合杂志》表格规则覆盖`
- `最终执行：单位必须写为（n，%）`

## Editing And Proofreading Consumption

V3 第一阶段要求：

- `editing` 基于同一套表格语义对象执行规则
- `proofreading` 基于同一套表格语义对象检查并报告

两者不能分别对表格做自己的“理解”。否则会出现：

- 编辑认为命中了
- 校对认为没命中
- 报告与执行不一致

因此，V3 的一个硬性原则是：

`同一张表在 editing 与 proofreading 中必须共享同一套表格语义层。`

## Knowledge Role

知识库在 V3 中仍然只承担投影与理解增强作用，例如：

- 保存典型三线表规则说明
- 保存期刊表格规范示例
- 保存表格对象解释文本

但知识库不能直接决定最终规则命中或执行。执行真源仍然只能来自规则库。

## Acceptance Standards

V3 第一阶段建议以如下 5 条作为完成验收：

1. 系统能把标准医学三线表稳定拆成语义对象，而不是只保留原始 Word 行列坐标。
2. 编辑模块能基于这些语义对象稳定命中表格规则，而不是依赖脆弱的字符串或固定坐标。
3. 校对模块能对同一语义对象输出可读检查结果。
4. 当期刊小模板存在覆盖规则时，系统能正确解释为什么最终执行的是期刊规则而不是通用规则。
5. 对一组代表性医学三线表样例，命中结果稳定、可复现、可回归测试。

## Done Definition

V3 第一阶段的“完成”，不以“自动排版多漂亮”为标准，而以以下 4 点同时成立为标准：

- 这张表已经先被系统“看懂”
- 规则已经稳定打到该打的位置
- 命中原因和覆盖原因人能看懂
- 同一张表在编辑和校对里用的是同一套语义层

只要这 4 点成立，V3 第一阶段就已经建立了后续自动修正、学习增强、更多期刊表格规则的低风险基础。

## Deferred To Later V3+ Work

以下内容建议留到 V3 后续阶段或更后版本：

- 表格复杂异常版式的大规模扩展识别
- 三线表自动修正与自动排版优化
- 表格规则学习闭环增强
- 命中质量大盘与批量运营分析
- PDF 表格语义层

## Summary

V3 第一阶段的本质不是“新增更多表格规则”，而是：

`建立 DOCX 医学三线表语义层，让规则真正能精准命中正确位置。`

只要这层底座建稳，大模板通用规则、期刊小模板覆盖规则、编辑执行、校对报告以及后续自动修正能力，都会建立在一个可复用、可解释、可扩展的基础之上。
