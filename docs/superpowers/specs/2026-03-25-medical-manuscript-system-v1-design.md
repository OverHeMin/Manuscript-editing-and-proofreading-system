# 医学稿件处理系统 V1 设计稿

**Date:** 2026-03-25
**Status:** Draft for review
**Purpose:** 打印版方案总稿，统一当前已确认的系统定位、边界、模板治理、工具分工与实施计划。

## 1. 产品定位

这是一个：

`面向医学稿件处理的 AI 工作平台`

它不是医学百科，不是通用问答产品，也不是允许模型静默自进化的黑盒系统。

V1 固定覆盖四大业务模块：

- `screening`：审稿 / 筛稿
- `editing`：编加 / 编辑
- `proofreading`：校对
- `knowledge`：医学知识库

另有一个受限子能力：

- `PDF 目录 - 正文 heading 一致性核对`

## 2. V1 核心边界

### 2.1 文件边界

V1 正式支持：

- `doc`
- `docx`

固定规则：

- `doc` 必须先统一转换为 `docx`
- 原始稿件永不覆盖
- 所有处理结果作为派生资产保存

PDF 边界：

- 不做通用 PDF 全文审校
- 只做目录与正文 heading 一致性核对

### 2.2 交付边界

V1 采用：

- 文件资产为正式结果真源
- 系统内支持在线预览、批注查看、协作评估
- `ONLYOFFICE DocumentServer` 进入 V1 正式评估范围
- 不把网页在线编辑器作为唯一正式稿件真源

### 2.3 生产态 AI 边界

生产态 AI 只能调用：

- `approved` 知识项
- `approved` 模板
- `approved` 规则
- `approved` prompt / checklist 资产

外部公开来源：

- 不直接进入生产证据链
- 只能用于审核员参考
- 只能用于知识草稿生成
- 只能用于学习候选提炼

所有 AI 结论必须具备：

- 证据引用
- 置信度
- 风险兜底说明

高风险医学问题默认保守，无法稳定判断时统一输出：

`需人工复核`

## 3. 业务输出规则

### 3.1 审稿模块

- AI 可以直接输出最终审稿结论和最终审稿报告
- 仍需记录证据引用、置信度、触发规则与审计信息
- 若案例要进入学习层，后续仍需人工选定为可学习案例

### 3.2 编加模块

- AI 可以直接输出最终编加稿件
- 系统保留原稿、标准化稿、编辑结果稿等版本资产
- 高风险内容或规则冲突时仍可转人工复核

### 3.3 校对模块

- AI 先输出草稿型校对结果
- 草稿包括结构化问题清单、Word 批注建议、校对报告草稿
- 必须经人工确认后，才能生成最终校对稿或最终批注版

### 3.4 处理模式

V1 采用模块独立调用：

- 用户上传稿件后，可直接发起审稿
- 可直接发起编加
- 可直接发起校对
- 可直接发起 PDF 一致性核对

不强制按 `审稿 -> 编加 -> 校对` 串联。

## 4. 稿件类型识别与选择

V1 采用双入口类型确定：

- 人工可在上传或处理前直接勾选稿件类型
- AI 也可自动识别并给出建议
- 若人工已明确勾选，则系统按勾选类型直接处理
- 若人工未勾选，则由 AI 识别并提示确认后处理

建议支持的固定稿件类型：

- `clinical_study`
- `review`
- `systematic_review`
- `meta_analysis`
- `case_report`
- `guideline_interpretation`
- `expert_consensus`
- `diagnostic_study`
- `basic_research`
- `nursing_study`
- `methodology_paper`
- `brief_report`
- `other`

V1 的模板与知识路由仍按主稿件类型走，不做多类型混合路由。

## 5. 稿件与文件资产模型

### 5.1 稿件模型

`Manuscript` 用于描述一篇稿件的业务主线，回答：

- 这篇稿件是谁上传的
- 标题、作者、稿件类型是什么
- 当前整体状态是什么
- 跑过哪些模块
- 各模块结果是什么
- 是否进入学习层

它是系统中的“主档案卡”。

### 5.2 文件资产模型

`DocumentAsset` 用于描述围绕稿件产生的所有文件版本，回答：

- 这是什么类型的文件
- 它从哪个文件派生而来
- 是原稿、标准化稿还是结果稿
- 是谁在什么时候生成的
- 当前是否为正式有效版本
- 是否能用于下载、预览、学习或审计

它是系统中的“文件版本库”。

### 5.3 建议资产类型

- `original`
- `normalized_docx`
- `screening_report`
- `edited_docx`
- `proofreading_draft_report`
- `proofread_annotated_docx`
- `pdf_consistency_report`
- `human_final_docx`
- `learning_snapshot_attachment`

### 5.4 为什么必须拆成两层

- `Manuscript` 管业务主线
- `DocumentAsset` 管文件版本和证据链

这样系统才能稳定回答：

- 哪个文件是 AI 产物
- 哪个文件是人工确认后的正式结果
- 哪个文件进入了学习层
- 哪个动作是谁做的

## 6. 组织、权限与协作模式

V1 是单机构内部系统，采用完整角色制：

- `admin`
- `screener`
- `editor`
- `proofreader`
- `knowledge_reviewer`
- `user`

协作模式：

- 不做复杂指派流
- 不做三段式工单流转
- 采用“权限可见 + 自主处理”的工作台模式

固定治理点：

- 只有 `admin` 可以发布模板
- `knowledge_reviewer` 负责知识项与学习候选审核

## 7. 模板治理系统

模板不是静态 prompt，而是可审核、可编辑、可绑定知识与经验资产的业务骨架。

### 7.1 总体组织方式

模板采用混合模式：

- 上层：按稿件类型建立 `TemplateFamily`
- 下层：在每个模板族下维护 `screening / editing / proofreading` 三套 `ModuleTemplate`

例如一个 `clinical_study` 模板族下包含：

- `clinical_study-screening`
- `clinical_study-editing`
- `clinical_study-proofreading`

这意味着：

- 新增一种稿件类型，本质上是新增一个模板族
- 在模板族内分别维护三个模块模板

### 7.2 模板版本治理

模板必须版本化，不能直接覆盖线上版本。

建议状态：

- `draft`
- `published`
- `archived`

建议流程：

1. 后台创建模板草稿
2. 编辑模板内容、知识绑定、规则和 prompt
3. 形成新版本草稿
4. 只有 `admin` 可以发布为 `published`
5. 旧版本保留可追溯记录

### 7.3 模板应包含的内容

每个 `ModuleTemplate` 至少包含：

- 模板名称
- 所属模板族
- 所属模块
- 适用稿件类型
- 版本号
- 状态
- 章节结构要求
- 输入要求
- 输出格式要求
- 风险规则
- 人工复核触发条件
- 主 prompt
- checklist
- 关联规则集
- 关联案例模式集
- 关联知识绑定

### 7.4 知识绑定

知识库中的 `approved` 知识项不直接写死进模板正文，而是通过 `TemplateKnowledgeBinding` 绑定到模板或章节。

建议支持的绑定用途：

- `required`
- `recommended`
- `risk_guardrail`
- `section_specific`

这样可以实现：

- 一个知识项挂到多个模板
- 一个模板绑定多个知识项
- 不同章节使用不同知识

### 7.5 学习经验如何回写模板

AI 学到的经验不能直接改 live 模板，只能先形成候选，再进入审核，再生成新模板版本。

正确链路是：

`案例完成 -> 进入学习候选 -> 审核通过 -> 生成模板新草稿版本 -> admin 发布`

学习候选可表现为：

- `template_update_candidate`
- `rule_candidate`
- `prompt_optimization_candidate`
- `checklist_update_candidate`

### 7.6 模板治理的价值

通过模板族、模块模板、知识绑定和学习回写，系统能持续扩展对更多医学稿件类型的稳定支持能力，而不依赖模型黑盒成长。

## 8. 学习层与知识治理

学习层不是模型训练层，而是：

`human-feedback-to-knowledge conversion`

学习最小颗粒度固定为：

`稿件类型 x 模块`

例如：

- `clinical_study-screening`
- `review-editing`
- `case_report-proofreading`

### 8.1 学习输入

- 原稿
- AI 输出
- 人工确认结论
- 人工最终稿
- 人工最终批注版
- 审核动作
- 审计日志中的高价值差异

### 8.2 学习前置门槛

进入学习层前必须满足：

- 已完成人工挑选或人工确认该案例具备学习价值
- 已补齐人工最终稿或人工最终批注版
- 已通过脱敏预检

不满足任一条件：

- 不生成正式学习快照
- 不进入学习候选流程

### 8.3 学习对象

至少包括：

- `ReviewedCaseSnapshot`
- `LearningRun`
- `LearningCandidate`

### 8.4 学习候选类型

- `rule_candidate`
- `case_pattern_candidate`
- `template_update_candidate`
- `prompt_optimization_candidate`
- `checklist_update_candidate`

### 8.5 学习候选状态

- `draft`
- `pending_review`
- `approved`
- `rejected`
- `archived`

### 8.6 审批责任

统一由：

- `knowledge_reviewer`

审批全部学习候选。

### 8.7 approved 回写

审核通过后，回写到对应 `稿件类型 x 模块` 方法资产包：

- `rules`
- `case_patterns`
- `templates`
- `prompts`
- `checklists`

### 8.8 自主学习的真实含义

系统的“自主学习”只允许表现为：

- 自动观察已审核案例
- 自动提炼候选
- 自动触发复盘任务
- 自动生成待审核建议

不允许：

- 自动发版到生产
- 自动改模型
- 自动绕过审核

## 9. 脱敏与合规

V1 采用：

- 学习链路强制脱敏预检
- 日常上传和日常处理不强制全拦截

凡是要进入以下对象，必须先跑脱敏检查：

- `ReviewedCaseSnapshot`
- `LearningCandidate`
- 知识草稿
- 长期复用资产池

推荐使用：

- `Presidio`

作为脱敏能力候选。

## 10. PDF 目录 - 正文一致性核对

V1 只支持这条最窄但高价值的链：

`上传/选择 PDF -> OCR 预处理 -> TOC 抽取 -> 正文 heading 抽取 -> heading 归一化 -> 一致性比较 -> 输出 issue 列表`

### 10.1 范围

只检查：

- TOC heading 是否在正文中存在
- heading 层级是否一致
- 编号是否一致
- 顺序是否一致
- 页码是否明显不一致
- 是否缺失、重复或跳序

### 10.2 issue 类型

- `toc_missing_in_body`
- `body_missing_in_toc`
- `toc_level_mismatch`
- `toc_numbering_mismatch`
- `toc_order_mismatch`
- `toc_page_mismatch`
- `needs_manual_review`

### 10.3 工具策略

- 普通可检索 PDF：直接抽文本结构
- 扫描 PDF：优先 `OCRmyPDF`
- 中文复杂页增强：`PaddleOCR`
- 学术结构增强位：预留 `GROBID` 适配器

## 11. 工具与功能服务角色

### 11.1 控制面固定分工

`superpowers`

- 需求澄清
- 方案设计
- spec 写作与维护
- implementation plan
- 决策日志
- 验收标准

`gstack`

- 工具调研
- 方案评审
- 页面流程验证
- 视觉与交互 QA
- 浏览器自动化检查
- 上线后巡检验证

`subagent`

- plan 批准后的任务拆分执行
- 明确边界内的实现任务
- 不改需求
- 不改总方案
- 不替代验收标准

### 11.2 运行面功能服务角色

不再增加新的同级总控角色，而是引入专职功能服务层：

`Temporal / Queue Orchestrator`

- 长任务编排
- 重试
- 超时
- 补偿逻辑

`ONLYOFFICE / Document Interaction Layer`

- 文档预览
- 批注查看
- 在线协作评估

`Label Studio / Human Review Bench`

- 知识审核
- 学习候选审核
- 分类标注沉淀

`Presidio / Privacy Gate`

- 学习链路前隐私风险过滤

`OCR Stack`

- `OCRmyPDF + PaddleOCR + GROBID`
- 负责抽取与结构识别

### 11.3 结论

- 作为总控分工，`gstack + superpowers + subagent` 已经够用，也是更稳的安排
- 真正需要补的是功能服务层，不是新的同级决策角色

## 15. 第一阶段优先事项

第一阶段目标定义为：

`先把“稿件接收 -> 标准化 -> 文件登记 -> 状态流转 -> 审计留痕”这条骨架主线做稳`

建议优先做这 5 个底座：

- 稿件与文件资产模型
- 统一 job 状态机
- 审计日志底座
- `doc -> docx` 主链路
- 角色权限

## 16. 本稿结论

这套 V1 方案的核心不是“让模型自己越来越懂所有医学稿件”，而是通过：

- 模板治理
- 知识审核
- 学习候选审核
- 文件与证据链管理
- 工具分层接入

构建一个可扩展、可审计、可持续变强的医学稿件处理系统。

## 17. 方案补强缺口清单

以下内容是当前方向已经明确、但仍建议补进后续 spec 的关键缺口。

### 17.1 高优先级缺口

#### 17.1.1 文档真源与版本规则

必须写清：

- `ONLYOFFICE`、下载文件、人工回传文件之间谁是正式真源
- 在线查看、批注、保存分别生成什么资产
- 是否允许覆盖旧版本

如果这个点不先写死，文档链路最容易混乱。

#### 17.1.2 模板、知识、规则、prompt 的优先级

必须明确冲突时谁优先，例如：

- 模板结构要求与知识规则冲突怎么办
- rule 与 checklist 冲突怎么办
- 低置信度时是否强制人工复核

建议单独做一张“决策优先级表”。

#### 17.1.3 知识过期、复审与失效机制

必须补：

- 知识项多久复审一次
- 指南更新后旧知识如何下线
- 模板绑定的知识过期后生产怎么处理

否则知识库会越积越多，但不一定越来越可靠。

#### 17.1.4 学习候选的准入、拒绝学习与去重机制

必须定义：

- 哪些案例禁止进入学习
- 哪些候选应自动归档
- 相似候选如何聚类、合并和排序

否则审核队列会快速膨胀。

#### 17.1.5 发布前评测与回归验证

模板、规则、prompt、知识一旦更新，建议先跑固定评测集再发布。

建议加发布门禁：

- 不过评测不允许上线

#### 17.1.6 失败回退与人工接管

要把主链路失败场景写清楚，例如：

- OCR 失败
- 类型识别冲突
- 模板缺失
- 外部服务超时
- ONLYOFFICE 不可用

这些情况分别怎么降级、怎么人工接管。

#### 17.1.7 审计日志的业务可读性

除底层事件日志外，还建议增加可读审计摘要，能回答：

- 这次结果为什么这样生成
- 用了哪些模板、知识、规则、证据
- 谁确认了什么

#### 17.1.8 校对模块的最终稿定义

虽然已明确“校对先草稿后确认再出最终稿”，但仍需补清最终稿到底包括：

- 最终批注版
- 最终问题清单
- 最终修订稿

中的哪些正式资产，避免模块理解不一致。

### 17.2 中优先级缺口

#### 17.2.1 稿件类型扩展策略

建议补“新增一种稿件类型时的标准流程”，例如：

- 新建模板族
- 配置模块模板
- 绑定知识
- 准备评测集

#### 17.2.2 模板覆盖率治理

建议定义一个稿件类型何时算“正式支持”，例如：

- 三个模块模板是否齐备
- 知识覆盖是否达标
- 评测通过率是否达标

#### 17.2.3 成本与资源治理

建议提前定义：

- 哪些任务同步
- 哪些任务异步
- 哪些结果缓存复用
- 哪些模型调用限流

#### 17.2.4 观察性与运维指标

建议补充：

- 任务成功率
- OCR 失败率
- 人工复核率
- 学习候选审批率
- 模板发布后回归通过率
- 各模块平均耗时

#### 17.2.5 数据保留与归档策略

建议定义：

- 原稿、派生稿、学习快照、审计日志分别保留多久
- 是否需要冷热分层
- 是否支持逻辑删除与归档恢复

#### 17.2.6 权限细化边界

虽然角色已定，但还建议补清：

- 谁能下载原稿
- 谁能看脱敏前内容
- 谁能看学习快照
- 谁能导出审核记录

#### 17.2.7 人工工作台设计

建议明确各角色首页要展示什么，例如：

- 待处理稿件
- 待审核候选
- 待发布模板
- 失败任务
- 风险提示

#### 17.2.8 导入与迁移能力

如果未来有历史模板、知识整理稿或老系统数据，建议预留导入策略，减少冷启动重复劳动。

### 17.3 建议新增的三个章节

建议在正式 spec 中新增：

- `质量与发布治理`
- `决策优先级与冲突处理`
- `运行失败与降级策略`

### 17.4 若只优先补五项，建议顺序

1. 文档真源与版本规则
2. 决策优先级与冲突处理
3. 知识过期与复审机制
4. 学习候选去重、拒绝学习与排序机制
5. 发布前评测与回归验证

## 18. 文档真源与版本规则

### 18.1 真源原则

系统中的正式真源永远是 `DocumentAsset`，不是浏览器页面，也不是 ONLYOFFICE 会话本身。

固定原则：

- 在线预览只是投影视图，不是正式真源
- 在线批注评估只是协作行为，不是最终资产本身
- 任何“保存”“确认”“导出”“发布”动作都必须生成或更新正式资产记录

### 18.2 资产写入规则

禁止直接覆盖旧资产内容。

固定规则：

- 原稿永不覆盖
- 标准化稿永不覆盖
- AI 结果稿默认生成新资产
- 人工确认结果生成新资产
- 导出行为不改变资产，只产生下载审计

### 18.3 ONLYOFFICE 与资产关系

ONLYOFFICE 在 V1 中定位为：

- 在线预览层
- 批注查看层
- 协作评估层

若发生会话内修改或批注固化，规则如下：

- 只允许写回为新的 `DocumentAsset` 版本
- 不允许直接回写覆盖旧版本
- 该新资产必须记录来源会话、操作者、时间、父资产和模块上下文

### 18.4 当前正式结果指针

每个稿件和每个模块结果都应保留“当前正式结果指针”，用于回答：

- 当前生效的审稿报告是哪一个
- 当前生效的编加稿是哪一个
- 当前生效的校对最终稿是哪一个

建议维护：

- `current_screening_asset_id`
- `current_editing_asset_id`
- `current_proofreading_asset_id`

### 18.5 校对模块最终稿定义

校对模块的最终正式结果建议固定为：

- `final_proof_issue_report`
- `final_proof_annotated_docx`

V1 默认不要求系统自动生成“最终修订正文”，除非人工另行上传确认后的修订稿。

## 19. 决策优先级与冲突处理

### 19.1 固定优先级

当多个来源同时参与决策时，优先级建议固定为：

1. 人工显式选择或人工确认
2. 硬性风险守卫与人工复核规则
3. 已发布模板的结构约束与模块边界
4. `approved` 知识项
5. `approved` 规则集
6. `approved` 案例模式
7. `approved` prompt 与 checklist
8. 模型自由推理

外部公开来源不进入生产优先级链。

### 19.2 冲突处理规则

当发生冲突时：

- 人工显式选择优先于 AI 识别
- 风险守卫优先于模板生成目标
- 模板结构约束优先于语言润色类建议
- 已批准知识优先于案例模式
- 案例模式与 prompt 不得推翻硬性知识规则

### 19.3 统一冲突出口

出现以下任一情况时，统一进入：

`需人工复核`

触发条件包括：

- 模板规则与知识规则冲突
- 多个知识项得出相反结论
- 稿件类型识别与人工选择冲突且未确认
- 关键判断置信度低于阈值
- 命中高风险医学或统计问题

### 19.4 决策留痕

每次生成结果时，系统应保留可追溯决策摘要，至少记录：

- 采用了哪个模板版本
- 命中了哪些知识项
- 命中了哪些规则或案例模式
- 哪些冲突被如何解决
- 是否触发人工复核

## 20. 知识生命周期治理

### 20.1 建议状态

除已有审核状态外，知识项建议支持更完整的生命周期：

- `draft`
- `pending_review`
- `approved`
- `deprecated`
- `superseded`
- `archived`

### 20.2 复审机制

知识项应按风险等级或来源类型定期复审。

建议默认复审周期可配置：

- 高风险知识：90 天
- 中风险知识：180 天
- 低风险知识：365 天

若机构另有要求，以机构策略为准。

### 20.3 失效机制

出现以下情况时，知识项应进入 `deprecated` 或 `superseded`：

- 权威来源已更新
- 原来源失效或不可追溯
- 发现与新指南明显冲突
- 内容重复且被更高质量版本替代

### 20.4 与模板的联动

若一个已发布模板绑定了失效知识，系统应自动触发：

- 模板复核提醒
- 风险提示
- 必要时阻止高风险生产调用

### 20.5 知识复用边界

知识项若过于具体、只适用于单个个案、无法抽象复用，则不得作为长期知识资产发布，只能保留为案例证据或学习参考。

## 21. 学习候选队列治理

### 21.1 学习准入规则

以下案例默认不进入学习：

- 证据链不完整
- 没有人工最终稿或最终批注版
- 脱敏未通过
- 属于明显一次性噪声或异常案例
- 医学事实结论本身仍存在争议且未形成稳定判断

### 21.2 候选打分维度

建议为每个候选维护基础评分，用于排序审核队列：

- 可复用性
- 证据强度
- 跨案例稳定性
- 风险等级
- 对模板或规则改进的预期收益

### 21.3 去重与聚类

系统应支持对候选进行相似聚类，至少按以下维度生成指纹：

- 稿件类型
- 模块
- 章节位置
- 问题类型
- 关联模板
- 关联知识标签

高相似候选应支持：

- 合并审阅
- 批量驳回
- 批量归档

### 21.4 自动归档规则

以下候选可自动降级或归档：

- 只出现一次且证据弱
- 完全重复已批准规则
- 仅针对某个具体个案、无法泛化
- 与现有高优先级知识或规则直接冲突

### 21.5 审核队列控制

为避免 `knowledge_reviewer` 队列爆炸，建议支持：

- 按评分排序
- 按模板族筛选
- 按稿件类型和模块筛选
- 批量处理
- 近似重复折叠显示

## 22. 质量与发布治理

### 22.1 发布对象

以下内容进入生产前都应经过发布治理：

- 模板新版本
- 知识项重大更新
- 规则集更新
- prompt 更新
- checklist 更新

### 22.2 评测集

每个 `稿件类型 x 模块` 建议维护一组固定评测集，至少包含：

- 标准样本
- 高风险样本
- 边界样本
- 历史易错样本

### 22.3 发布门禁

建议设定以下门禁：

- 结构校验通过
- 依赖资产完整
- 评测集达标
- 高风险样本无新增回归
- 审计与变更说明完整

不满足条件则不得发布。

### 22.4 模板发布流程

建议流程如下：

1. 生成新模板草稿版本
2. 补齐知识绑定、规则、prompt、checklist
3. 跑评测集
4. 生成变更摘要
5. 由 `admin` 发布

### 22.5 回滚机制

每次发布都必须支持版本回滚。

回滚时应：

- 恢复到上一稳定版本
- 保留本次失败发布记录
- 自动生成回滚审计事件

## 23. 运行失败与降级策略

### 23.1 类型识别失败

若 AI 无法稳定识别稿件类型：

- 提示人工选择
- 不直接进入正式处理

### 23.2 模板缺失

若缺少对应 `稿件类型 x 模块` 的已发布模板：

- 不允许按该模板正式生产
- 可选择人工改用 `other` 模板族
- 或交由 `admin` 补模板后再处理

### 23.3 OCR 失败

若 OCR 质量不足或 PDF 结构混乱：

- 输出 `needs_manual_review`
- 保留失败原因和错误证据
- 不继续给出高置信度结构判断

### 23.4 外部服务不可用

若 ONLYOFFICE、OCR、向量检索或其他外部服务不可用：

- 优先降级到可用主链路
- 无法安全降级时，转人工复核
- 所有降级动作必须进入审计

### 23.5 模型调用失败

模型超时、失败或结果不完整时：

- 自动重试有限次数
- 超限后进入人工接管或任务失败队列
- 不允许输出伪装成成功的空结果

## 24. 成本、性能与资源治理

### 24.1 任务同步 / 异步边界

建议默认：

- 上传基础校验：同步
- `doc -> docx`、OCR、解析、AI 处理：异步
- 结果查询与下载：同步读取

### 24.2 缓存与复用

建议缓存：

- 稿件类型识别结果
- 文档结构抽取结果
- PDF OCR 结果
- 知识检索结果摘要

但缓存必须绑定：

- 输入版本
- 模板版本
- 知识版本

任何关键版本变化都应导致缓存失效。

### 24.3 模块优先级

建议运行优先级可配置：

- 用户发起的正式处理任务优先
- 预览和低成本查询次之
- 学习复盘任务再次之
- 批量历史补处理最低

### 24.4 限流与配额

建议为以下能力设置限流或配额：

- OCR
- 大模型调用
- 批量学习任务
- 大文件上传

## 25. 观察性与运维指标

建议至少跟踪以下指标：

- 稿件上传成功率
- `doc -> docx` 转换成功率
- OCR 成功率
- 各模块任务成功率
- 人工复核率
- PDF 一致性核对失败率
- 学习候选生成率
- 学习候选审批通过率
- 模板发布后回归通过率
- 各模块平均耗时

建议同时保留：

- 技术监控视图
- 业务运营视图

## 26. 数据保留与归档策略

数据保留应按机构策略可配置，但建议系统支持分层保留：

- 原稿与正式结果长期保留
- 中间处理产物按策略保留
- OCR 临时文件和短期缓存定期清理
- 被拒绝学习候选和中间快照可采用更短保留期

建议系统支持：

- 逻辑删除
- 归档恢复
- 按资产类型分级存储

## 27. 权限细化边界与人工工作台

### 27.1 权限细化

除角色定义外，建议进一步明确：

- 谁能下载原稿
- 谁能查看脱敏前内容
- 谁能查看学习快照
- 谁能导出审计和审核记录
- 谁能查看模板草稿与已发布模板差异

### 27.2 人工工作台

建议不同角色首页重点不同：

`admin`

- 系统状态
- 待发布模板
- 失败任务
- 高风险告警

`knowledge_reviewer`

- 待审核知识
- 待审核学习候选
- 高优先级候选聚类

`screener / editor / proofreader`

- 待处理稿件
- 最近结果
- 风险提示
- 人工复核待办

`user`

- 我的稿件
- 最近结果
- 下载记录

## 28. 稿件类型扩展、覆盖率与迁移策略

### 28.1 新稿件类型扩展流程

新增一种稿件类型时，建议固定流程：

1. 建立新的 `TemplateFamily`
2. 补齐 `screening / editing / proofreading` 模板
3. 绑定基础知识
4. 建立评测样本
5. 通过评测后再标记为正式支持

### 28.2 覆盖率治理

建议将支持程度分层：

- `experimental`
- `beta`
- `fully_supported`

达到 `fully_supported` 至少需满足：

- 三个模块模板齐备
- 核心知识覆盖达标
- 评测集通过
- 审计和人工复核链路完整

### 28.3 导入与迁移能力

建议预留以下导入能力：

- 历史模板导入
- 历史知识清单导入
- 历史案例批量入库
- 历史文件资产登记

导入数据默认不直接进入生产，应先进入：

- 待审核区
- 待清洗区
- 待映射区

## 29. 前端交互与信息架构

### 29.1 设计目标

前端不只是展示结果，而是承担：

- 多角色工作台
- 稿件处理主流程
- 模板治理
- 知识审核
- 学习候选审核
- 风险提示与人工复核

设计目标应包括：

- 信息层级清晰
- 风险状态一眼可见
- 人工操作最少跳转
- 文件与结果版本容易追溯
- 桌面端优先，移动端按场景扩展

### 29.2 建议主导航

建议 Web 端一级导航包括：

- 工作台
- 稿件
- 审稿
- 编加
- 校对
- PDF 核对
- 知识库
- 模板治理
- 学习审核
- 审计
- 系统设置

### 29.3 角色工作台

`admin`

- 系统运行状态
- 待发布模板
- 失败任务
- 高风险告警
- 平台配置摘要

`knowledge_reviewer`

- 待审核知识
- 待审核学习候选
- 相似候选聚类
- 近期驳回和归档记录

`screener`

- 待审稿件
- 最近审稿结果
- 高风险待复核项

`editor`

- 待编加稿件
- 最近输出稿件
- 模板变更提示

`proofreader`

- 待校对稿件
- 待人工确认草稿
- PDF 核对异常项

`user`

- 我的稿件
- 当前状态
- 最近结果与下载记录

### 29.4 核心交互流程

应重点设计以下流程：

- 上传稿件并选择稿件类型
- AI 识别类型并人工确认
- 审稿 / 编加 / 校对独立发起
- ONLYOFFICE 中预览与批注评估
- 人工确认校对结果
- 学习候选审核
- 模板发布
- 知识入库审核
- PDF 一致性异常处理

### 29.5 风险交互

高风险或冲突情况必须有统一交互：

- 明确风险标签
- 明确证据来源
- 明确为什么进入人工复核
- 明确下一步可执行动作

### 29.6 移动端策略

Web 是主工作端。

移动端优先承接：

- 知识审核
- 轻量查询
- 审批动作

不建议在手机端承接复杂文档编辑与全文处理主流程。

## 30. 交付、升级与平台迁移策略

### 30.1 推荐交付形态

建议按以下形态规划：

- 本地开发：单机开发环境
- 测试环境：Docker Compose
- 生产环境：优先 Linux 服务器，可扩展到云主机或容器平台

### 30.2 环境分层

建议固定环境：

- `dev`
- `staging`
- `prod`

不同环境必须分离：

- 数据库
- 对象存储
- 模板发布配置
- 外部服务配置

### 30.3 配置管理

所有依赖必须配置化，不允许写死在代码中：

- 数据库连接
- Redis
- 对象存储
- OCR 服务
- ONLYOFFICE
- LLM 服务
- 日志与监控地址

### 30.4 平台迁移原则

系统必须支持从当前机器迁移到其他服务器或平台。

固定原则：

- 应用与配置分离
- 资产存对象存储，不依赖固定本地路径
- 数据库 schema 全部走 migration
- 模板、知识、prompt、规则支持导出与导入
- 服务地址全部配置化

### 30.5 升级策略

每次版本升级应包括：

- 应用版本升级
- 数据库 migration
- 配置兼容检查
- 模板与知识版本兼容检查
- 回滚预案

### 30.6 备份与恢复

必须定义备份对象：

- Postgres 数据
- MinIO / 对象存储
- 模板与知识资产
- 审计日志
- 配置与密钥引用关系

恢复时应支持：

- 整体恢复
- 单稿件恢复
- 单资产恢复

## 31. 远程开发与维护工作流

### 31.1 原则

Codex 可以作为维护助手，但不应成为系统运行依赖。

更稳的原则是：

- 系统运行依赖标准服务
- 代码维护依赖 Git 工作流
- Codex 参与设计、编码、review、验证和文档维护

### 31.2 推荐维护方式

建议采用：

- Git 仓库统一托管代码
- 远程开发机或服务器作为维护环境
- 通过 SSH / VPN / 堡垒机访问
- 通过分支、PR、测试和发布流程上线

### 31.3 Codex 在远程维护中的角色

Codex 可用于：

- 读代码
- 写代码
- 写 spec
- review 设计和实现
- 跑验证
- 更新文档

但不建议：

- 直接在线修改生产代码
- 跳过仓库流程直接改线上文件

### 31.4 维护安全要求

远程维护必须满足：

- 独立账号
- 最小权限
- 操作审计
- 生产只读优先
- 变更必须可追溯

### 31.5 远程维护与部署分离

建议区分：

- 远程开发环境
- 测试环境
- 生产环境

避免“在生产机上直接开发”的模式。

## 32. 认证、密码与安全治理

### 32.1 认证基本原则

认证与安全应作为一级系统能力，而不是上线前补丁。

### 32.2 密码安全

必须满足：

- 密码不明文存储
- 使用强哈希算法，如 `Argon2id` 或 `bcrypt`
- 支持密码复杂度规则
- 支持密码重置与失效策略

### 32.3 登录与会话安全

建议支持：

- 登录失败次数限制
- 会话超时
- 刷新机制
- 关键设备重新验证
- 管理员更严格的会话策略

### 32.4 管理员账号保护

建议为管理员预留：

- 二次确认
- 更严格密码策略
- 后续 MFA 扩展位

### 32.5 敏感操作保护

以下操作建议增加二次确认或高等级审计：

- 发布模板
- 批量审批知识或学习候选
- 删除稿件
- 导出敏感记录
- 调整系统配置

### 32.6 密钥与配置安全

必须遵守：

- 密钥不写入代码库
- 环境变量或密钥管理系统保存敏感配置
- 第三方服务 token 单独管理
- 不同环境使用不同密钥

### 32.7 安全审计

必须审计：

- 登录
- 登出
- 密码重置
- 权限变更
- 模板发布
- 知识审核
- 学习候选审批
- 敏感数据导出

## 33. 多端渠道扩展策略

### 33.1 总体原则

V1 主交付仍是 Web。

多端扩展必须复用统一后端能力，不允许：

- 小程序单独维护一套知识逻辑
- 小程序单独维护一套审核状态机

### 33.2 微信小程序定位

微信小程序优先承接：

- 知识审核入口
- 知识查询
- 审核意见提交
- 审核历史查看

不建议首期承接：

- 审稿全文处理
- 编加全文处理
- 校对全文处理
- ONLYOFFICE 类复杂文档交互

### 33.3 小程序首期能力建议

建议首期只做：

- 待审核知识列表
- 知识详情
- 审核通过 / 驳回
- 审核备注
- 已审核记录查询

### 33.4 API 复用要求

为支持小程序，知识库模块 API 应提前满足：

- 认证可移动端复用
- 审核动作有清晰接口
- 列表查询支持分页与筛选
- 权限控制与 Web 端一致

### 33.5 后续扩展

若小程序运行稳定，后续可逐步扩展：

- 学习候选轻审核
- 风险提醒查看
- 模板发布提醒只读通知

但不建议让小程序承接复杂文档处理主流程。

## 34. AI 模型治理与路由

### 34.1 总体原则

系统必须支持后台切换 AI 提供方和模型，但模型切换应是受治理的能力，而不是任意任务中的临时自由选择。

### 34.2 模型目录

建议维护统一的 `ModelRegistry`，每条记录至少包括：

- provider 名称
- model 名称
- 版本标识
- 适用模块
- 是否允许生产使用
- 成本信息
- 限流信息
- 备注与评测结论

### 34.3 模型选择层级

建议按以下层级生效：

1. 系统默认模型
2. 模块默认模型
3. 模板覆盖模型
4. 任务级临时指定模型

但任务级指定只能从已批准模型池中选择。

### 34.4 模型切换治理

生产模型切换必须满足：

- 已进入 `ModelRegistry`
- 已通过基础评测
- 已通过对应模块或模板评测
- 已保留回滚目标模型
- 已生成切换审计

### 34.5 fallback 与灰度

建议支持：

- 主模型失败时切备用模型
- 新模型小流量灰度
- 模型间对照评测

### 34.6 审计要求

每次 AI 调用至少应记录：

- provider
- model
- model version
- 模块
- 模板版本
- 任务 ID
- 耗时
- token / 成本摘要

## 35. 扩展性与模块化接口

### 35.1 总体原则

新增模块或功能不应破坏现有底座，必须通过统一合同接入系统。

### 35.2 建议预留的统一合同

`Module Registry`

- 注册模块名称
- 模块权限
- 菜单入口
- 任务类型
- 输出资产类型

`Job Contract`

- 每个模块都走统一任务状态机

`Asset Contract`

- 每个模块输出都走统一 `DocumentAsset`

`Template Contract`

- 使用模板的模块必须接入统一模板治理

`Audit Contract`

- 新模块必须自动写审计

`AI Contract`

- 新模块若调用 AI，必须走统一模型网关

`Review Contract`

- 新模块若需要人工确认，必须接统一复核机制

### 35.3 功能扩展原则

新增功能时，优先通过：

- 新模板族
- 新模块模板
- 新任务类型
- 新资产类型
- 新工作台入口

来扩展，而不是直接修改现有模块的隐含逻辑。

## 42. 医学知识库架构、调用与检索

### 42.1 总体原则

医学知识库是系统的关键底座，不应只是“存一堆文本”，而应是：

- 可审核
- 可检索
- 可绑定模板
- 可被模块精准调用
- 可被后续扩展持续治理

### 42.2 是否需要添加、编辑知识库功能

必须需要，而且应是后台一级能力。

建议至少支持：

- 新建知识草稿
- 编辑知识草稿
- 已批准知识生成新版本草稿
- 审核通过 / 驳回 / 归档
- 批量导入
- 历史版本查看
- 与模板绑定

固定原则：

- 已批准知识不应直接原地覆盖编辑
- 修改应生成新版本草稿

### 42.3 建议知识项结构

每个知识项至少应包含：

- 标题
- 标准化陈述
- 摘要
- `knowledge_kind`
- `module_scope`
- 适用稿件类型
- 适用章节
- 风险标签
- 学科标签
- 证据级别
- 来源类型
- 来源链接
- 状态
- 生效时间
- 失效时间
- 关联模板
- 同义词或术语别名

### 42.4 审稿、编加、校对如何准确调用知识

建议采用“上下文路由 + 结构过滤 + 混合检索 + 重排序”四层机制。

第一层：任务上下文路由

从当前任务中提取：

- 当前模块
- 稿件类型
- 章节位置
- 当前模板版本
- 风险标签
- 问题类型

第二层：结构化过滤

只保留满足以下条件的知识：

- `approved`
- 未失效
- `module_scope` 匹配
- 稿件类型匹配或通用
- 章节匹配
- 风险标签匹配

第三层：混合检索

同时使用：

- 精确标签匹配
- 全文检索
- 向量检索

第四层：重排序

按以下因素排序：

- 是否与当前模板显式绑定
- 证据级别
- 来源优先级
- 风险守卫权重
- 章节贴合度
- 历史命中效果

### 42.5 模块调用建议

`screening`

优先调用：

- 研究设计规则
- 伦理合规知识
- 统计学风险守卫
- 稿件结构完整性要求

`editing`

优先调用：

- 结构模板要求
- 栏目命名规范
- 摘要与章节写作规范
- 术语统一规则

`proofreading`

优先调用：

- 医学术语规范
- 单位与统计表达规则
- 前后一致性检查规则
- 风险提示知识

### 42.6 如何定位庞大知识库中的局部数据

建议同时建立以下能力：

- 分层分类树
- 标签体系
- 同义词与术语归一化
- Faceted 搜索
- 向量检索
- 全文检索
- 模板绑定视图
- 稿件类型视图
- 风险标签视图

后台应允许按这些维度快速筛选：

- 模块
- 稿件类型
- 章节
- 风险等级
- 来源类型
- 证据级别
- 状态
- 关联模板

### 42.7 知识包与模板绑定

建议为每个模板族和模块模板形成“知识包”概念。

也就是：

- 一个模板默认绑定一组核心知识
- 执行时先召回模板绑定知识包
- 再补充动态检索结果

这样能显著提高调用准确率，避免在庞大知识库中盲检索。

### 42.8 去重与失效治理

知识库规模变大后，必须支持：

- 重复知识检测
- 过期知识标记
- 被替代知识标记
- 冲突知识提示

### 42.9 人工维护界面

建议后台至少提供：

- 知识列表页
- 知识详情页
- 知识编辑页
- 知识审核页
- 知识绑定模板页
- 知识版本对比页
- 知识搜索与筛选页
