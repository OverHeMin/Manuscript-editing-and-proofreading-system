# 开发与 Review 检查清单

## 1. 使用方式

本文档用于以下场景：

- 开始一项新开发前，自查边界是否清楚
- 提交代码前，自查是否达到可 review 状态
- 人工 review 时，统一检查重点
- Phase 收口前，确认是否具备进入下一阶段的条件

建议方式：

- 小改动至少执行“提交前检查”和“review 检查”
- 涉及业务规则、AI 逻辑、知识库、稿件资产的改动，四个阶段都过一遍
- 每次 review 都以本清单为共同语言，减少漏项和反复返工

## 2. 开发前检查

开始写代码前，先确认：

- 这次改动属于哪个明确模块，是否会跨模块影响
- 改动是否触及 `screening`、`editing`、`proofreading` 中的业务差异
- 是否会影响校对“草稿 -> 人工确认 -> 最终稿”链路
- 是否会影响 `DocumentAsset` 权威资产链路
- 是否会影响模板、知识、学习、模型、运行时权限中的任一治理对象
- 是否需要同步更新文档、接口、类型契约或管理后台入口
- 是否已经知道应该补什么测试，而不是写完再想

如果以上任一问题回答不清楚，不应直接进入实现。

## 3. 提交前检查

提交代码前逐项确认：

- 命名是否能直接体现业务含义，而不是依赖上下文猜测
- 复杂规则是否写了高价值注释，解释“为什么这样做”
- 逻辑是否被拆到合适层级，没有把业务规则塞进控制器、路由或页面壳里
- 没有把多个不相关改动混入同一次提交
- 相关测试、类型检查、smoke 或文档校验已经执行
- 如果没有执行完整验证，是否明确记录了原因和影响范围
- 文档入口是否同步更新，后续维护者能找到规则来源

重点自查：

- 审稿模块是否仍允许直接输出最终稿
- 编加模块是否仍允许直接输出最终稿
- 校对模块是否仍保持先草稿、再确认、后终稿
- 知识库调用是否仍可定位、可审计、可限定范围
- 管理员能力是否没有错误暴露给业务角色

## 4. Review 检查

review 时至少检查以下问题：

- 改动是否真正解决了当前任务，而不是顺手扩散了额外范围
- 业务语义是否正确，是否符合医学稿件处理场景
- 模板、知识、学习三者关系是否仍然清晰
- AI 模型切换、运行时治理、工具权限是否仍有明确管理员边界
- 是否破坏既有稿件生命周期、状态机或审计追踪
- 是否存在“看似复用，实际把不同模块语义揉在一起”的情况
- 测试是否覆盖了最容易返工的分支，而不只是 happy path
- 注释和文档是否解释了未来最容易误解的地方

发现以下情况时，应视为高风险：

- 规则写在多个地方，但没有单一真源
- 草稿与最终稿逻辑混淆
- 知识命中结果不可解释
- 学习沉淀可直接覆盖正式模板而缺少审核
- 运行时工具权限过宽
- 文档资产写回链路不清楚

## 5. 医学稿件系统专项检查

### 5.1 稿件与资产

- 原始稿件、规范化稿件、草稿、最终稿是否都能区分清楚
- 是否仍保持权威资产导出规则
- 是否有可能把预览层、临时文件、会话链接误当正式资产

### 5.2 模板与知识

- 模板是否支持后台创建、编辑、版本化
- 知识是否能定位到具体模板，而不是全局漂浮
- 学习经验是否经过治理后才能沉淀到模板
- 知识审核状态是否影响实际调用结果

### 5.3 审稿 / 编加 / 校对

- 审稿规则没有被编加或校对逻辑污染
- 编加规则没有覆盖掉审稿判断
- 校对规则没有绕过人工确认直接终稿
- 固定稿件类型和人工勾选稿件类型是否仍能共存

### 5.4 Agent Runtime 与工具治理

- `ModuleExecutionProfile` 与 `Agent Runtime` 是否仍分层清晰
- `SandboxProfile`、`RuntimeBinding`、`ToolPermissionPolicy` 是否保持独立职责
- 运行时日志、验证结果、评估结果是否可追踪
- `gstack`、`superpowers`、`subagent` 等后台能力是否仍只对管理员开放

## 6. Phase 收口检查

在一个 Phase 准备收口或进入下个 Phase 前，至少确认：

- 当前 Phase 范围内的功能、文档、测试、治理对象已经闭环
- 还未完成的内容是否明确记录，而不是隐性留坑
- 迁移、远程维护、版本升级、后续扩展是否保留了接口或文档说明
- 本 Phase 新增的规则是否已写入可查文档
- 下一 Phase 的工作不会被当前阶段的模糊边界拖累

## 7. 建议的 Review 输出格式

如果后续要形成统一 review 习惯，建议按以下顺序输出：

1. 发现的问题
2. 风险与边界
3. 已验证内容
4. 仍未验证内容
5. 是否建议进入下一步

这样比“整体看着没问题”更容易积累长期维护证据。
