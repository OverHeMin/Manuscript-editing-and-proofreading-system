# 代码质量与 Review 基线

## 1. 目标

本文档定义医学稿件处理系统 V1 的默认开发质量约束，用于降低以下风险：

- 业务规则写散后难以 review
- AI 相关逻辑缺少边界说明，后期难以定位问题
- 缺少测试与验证证据，导致“看起来能跑”但后期返工
- 多阶段持续扩展后，模块职责混乱、维护成本上升

默认原则：

- 先保证可读，再追求抽象
- 先把业务边界写清楚，再扩展能力
- 先有验证证据，再声称完成
- 先控制长期维护成本，再追求短期编码速度

## 2. 注释策略

本项目要求写“高价值注释”，不要求写“重复代码表面含义”的注释。

必须优先加注释的场景：

- 稿件生命周期状态流转
- 审稿、编加、校对三类模块的差异性规则
- 校对“草稿 -> 人工确认 -> 最终稿”的特殊流程
- 知识库命中、模板绑定、学习沉淀的判定逻辑
- AI 模型路由、Agent Runtime、Tool Gateway 的权限边界
- 文档资产转换、权威文件写回、外部工具集成边界
- 一段代码中存在不直观的取舍、回滚、兼容策略

注释应该解释：

- 为什么这样设计
- 这段逻辑在保护什么业务约束
- 哪些输入或状态不能被随意改变
- 后续扩展时需要保持什么兼容性

注释不应该做的事：

- 重复代码字面意思
- 为简单赋值或简单判断写解释
- 用大段注释掩盖函数过长、命名含糊的问题
- 写与实际行为容易失效的“未来计划式注释”

## 3. 命名与结构约束

### 3.1 命名

- 优先使用业务词汇命名，而不是技术缩写
- 一个名称只表达一种职责，避免“manager”“helper”这类泛化名称
- 重要状态、类型、策略优先通过枚举、联合类型、契约对象表达
- 避免把同一个业务概念在不同模块里起多个近义名字

### 3.2 结构

- 同一条业务规则应尽量集中在一个可定位的模块中维护
- 复杂流程拆成可单测的小函数，不把全部逻辑堆在控制器或路由层
- 外部依赖边界明确分层：
  - 领域规则
  - 基础设施适配
  - 管理后台编排
  - 前端展示与操作壳
- 不允许把“临时兼容逻辑”悄悄扩散成长期主流程

### 3.3 本项目的重点边界

- `screening`、`editing`、`proofreading` 只承载各自业务语义，不能互相侵入
- `ModuleExecutionProfile` 负责业务执行上下文
- `Agent Runtime` 负责运行时治理，不能反向吞并业务层语义
- `DocumentAsset` 及其衍生资产必须维持权威链路
- 模板、知识、学习、模型、工具权限都必须可审计、可版本化、可回溯

## 4. 测试与验证基线

### 4.1 什么时候必须补测试

以下改动默认必须补充或更新测试：

- 新增业务规则
- 修改状态流转
- 修改模板绑定或知识命中逻辑
- 修改 AI 路由、权限、运行时注册表
- 修改稿件资产写回、导出、草稿/终稿关联逻辑
- 修复 bug，尤其是已经出现过返工的问题

### 4.2 测试分层

- 合同测试：验证跨包类型、数据结构、约束对象
- 服务级测试：验证注册表、策略判断、状态流转、规则选择
- 流程级测试：验证上传、治理、执行、学习沉淀等关键链路
- Smoke 验证：验证 API / Web / Worker 基线可启动

### 4.3 对“已完成”的证据要求

默认至少要给出与改动范围匹配的验证证据：

- 文档或配置改动：确认入口文档和交叉引用没有遗漏
- 单模块代码改动：跑对应测试或类型检查
- 跨模块改动：跑相关模块测试，再补至少一轮全局校验的一部分
- 发布前或阶段收口：跑 `pnpm lint`、`pnpm typecheck`、`pnpm test` 或说明未执行原因

不能用以下方式替代验证：

- “代码看起来没问题”
- “上次跑过应该还能过”
- “这个改动很小应该不用测”

## 5. Review 清单

每次 review 至少检查以下问题：

- 业务语义是否准确，是否把医学稿件场景说清楚
- 审稿、编加、校对差异是否被正确保留
- 校对模块是否仍保持“先草稿、再确认、后终稿”
- 知识库调用是否可解释、可回溯、可限定范围
- 新逻辑是否破坏了模板、知识、学习三者之间的治理边界
- AI 运行时、模型切换、工具权限是否仍在管理员控制之下
- 命名是否能帮助后来者快速读懂，而不是增加理解成本
- 是否补了必要注释、测试和文档入口

## 6. 返工高风险区域

以下区域是本项目最容易因理解偏差而返工的地方，改动时要提高门槛：

- Word / PDF / OCR / 结构抽取之间的权威资产转换
- 草稿稿件与最终稿件的映射关系
- 学习候选、已采纳经验、模板固化经验之间的边界
- 知识召回、模板知识定位、知识审核状态
- 模型切换后行为差异与评估机制
- Tool Gateway、Sandbox、RuntimeBinding 等治理对象
- 微信小程序、后台管理端、主业务端之间的权限与数据暴露边界

这些区域的改动建议同时具备：

- 代码注释
- 测试覆盖
- 文档说明
- 审计或日志字段的可追踪性

## 7. 提交与变更粒度

- 一个提交尽量只解决一类问题
- 大改动按任务拆分，避免把“新功能 + 重构 + 修 bug + 文档补丁”混在一起
- 如果改动会影响未来迁移、升级、远程维护，必须同步更新相关文档
- 不把未验证的“顺手优化”混入当前任务

## 8. 完成定义

一个改动要被视为“完成”，至少应满足：

- 代码能被读懂，命名与结构没有明显歧义
- 关键业务边界有足够注释或文档说明
- 与改动范围相匹配的测试或验证已经执行
- 文档入口没有遗漏，后续维护者能找到规则来源
- 没有把本应管理员治理的能力暴露给业务角色

如果以上条件不满足，应视为“部分完成”，不能直接当成稳定成果进入下一阶段。

## 9. 本项目默认执行方式

后续开发默认采用以下做法：

- 复杂业务逻辑写清楚“为什么”
- 关键模块改动优先补测试
- 完成前必须重新验证，不凭印象判断
- 重要规则同步写进文档，不只留在对话里
- 发现会影响长期维护成本的问题，优先现在修正，不留到后期集中返工
